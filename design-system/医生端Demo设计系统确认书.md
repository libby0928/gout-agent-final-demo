## 〇、v2 视觉基线变更说明（必读）

v1 基于旧策略「蓝色工作台、刻意区别于患者端」撰写（主色 `#0052d9`、低圆角、紧凑行高、灰底）。**用户在 v1 产出期间改了核心视觉决策**，新方向为：

> **医生端整体向患者端视觉靠拢**——主色、渐变、圆角、行高、装饰氛围全部向患者端统一，采用青绿→蓝渐变 `#03d49f → #01a1ff`（即 `tokens.css` 的 `--patient-gradient` / `--patient-primary` / `--patient-primary-2`）。两端视觉基本同调性。

**仍保留的医生端特质（不可丢失）**：
- 信息密度高于患者端（紧凑列表、分块、来源标记更密，但可用大圆角承载）
- 底部用操作条（`.ds-doctor-actionbar`）而非患者端四胶囊浮动导航
- 状态色语义铁律不变：橙=待确认/待处理、绿=已确认/已完成、红=异常/风险/删除
- AI 辅助卡仍必须标注「诊疗辅助建议」标题（依据蓝条/风险红条/建议绿条）
- 线性 SVG outline 图标仍保留（比患者端圆角彩色方块更克制专业）

本确认书以下所有章节均按 v2 新基线撰写，覆盖 v1。

## 一、复用声明（Reuse Statement）

本确认书**不另起炉灶**。医生端 Demo 直接复用以下资产，原型构建师应优先照抄，禁止重写底层变量或重新定义视觉语言。

| 复用对象 | 路径 | 复用范围 |
|---------|------|---------|
| 共享变量 | `design-system/tokens.css` | **主色/渐变/圆角/字号/行高/装饰直接复用 `--patient-*` 变量**；`--doctor-*` 系列仅保留作状态色 soft 底等辅助用途 |
| 患者端展示外壳模板 | `design-system/templates/iphone-showcase-patient.html` | 外壳四层结构（背景/投影/机身/内屏） |
| 全局 Demo 规范 | `docs/03-原型与规范/痛风智能体Demo设计规范_AI执行版.md` | 五·医生端 / 七·Demo 导航 / 八·交互与状态联动 |

### 1.1 主色与视觉语言切换（v2 核心）
- **医生端主色 = 患者端青绿蓝渐变**：`--patient-gradient`（`linear-gradient(234deg, #03d49f 0%, #01a1ff 100%)`）、`--patient-primary`（`#03d49f`）、`--patient-primary-2`（`#01a1ff`）。
- `--doctor-primary`（旧蓝 `#0052d9`）**不再作为医生端主色**。原 `--doctor-*` 系列降级为辅助用途：状态色 soft 底（`--doctor-orange-soft` / `--doctor-green-soft` / `--doctor-red-soft`）、高密度场景的灰底/描边（如需）仍可引用 `--doctor-line` 等中性变量；语义状态色本身（`--doctor-orange` / `--doctor-green` / `--doctor-red`）保留为状态语义锚点，与 v1 一致。
- 圆角体系改用患者端大圆角：卡片 18–26px、按钮 20–25px（参考 `--radius-xl` 22px / `--radius-2xl` 26px，新增按钮可用 `--radius-pill` 胶囊形）。
- 行高改用患者端宽松档（1.55–1.6），但高密度列表行内仍可局部紧凑（不低于 1.4）。
- 装饰氛围复用患者端：浅青绿到浅蓝渐变背景、轻阴影、轻描边，允许柔光/圆形装饰（不再禁用）。

> ⚠️ 重要：q-5 要求「采集屏模拟交互——语音接诊/补充资料以展示采集状态+提取结果回填为主，不做真实音频/识别」。因此 P2 四屏（语音接诊、补充资料 OCR、就诊码、主动建档）是**模拟态演示屏**，不涉及真实音频/视频/识别算法，只做状态展示与结果回填动画。

## 二、红线清单（Hard Constraints / Do-Never）

以下为不可违反的设计与语义红线，违反任意一条即判为不合格交付。

### 2.1 视觉红线（v2 重写）
1. **医生端不得退化为纯患者端翻版**：必须保留医生端特质——高密度列表、底部操作条、来源标记六类、状态色语义铁律、AI 卡标注「诊疗辅助建议」、线性 SVG outline 图标。两端同调性但医生端信息密度明显更高。
2. **主色须为青绿蓝渐变**：顶部栏、主按钮、关键强调使用 `--patient-gradient`，**不得**回退到旧蓝 `#0052d9` 作主色。
3. **禁 emoji / 单字图标**：所有功能图标使用 SVG 线性图标（stroke 1.5–2px，Lucide / Heroicons outline 风格）。**禁止**用 emoji（✅⚠️📱）或单字（"扫""确""返""接"）代替功能图标。
   - 例外：演示导航分组缩写、流程编号、状态缩写等非正式展示文案可保留文字（见 §七 导航映射）。
4. **禁患者端四胶囊底部导航**：医生端底部用 `.ds-doctor-actionbar` 操作条（按页面定制），不用患者端 `ds-mobile-tabs`。

### 2.2 语义红线（保留 v1）
5. **状态色语义严格**：
   - 橙色 `--doctor-orange` **仅**用于「待确认/待处理/提醒」语义。
   - 绿色 `--doctor-green` **仅**用于「已确认/已完成」状态标记。
   - 红色 `--doctor-red` **仅**用于「异常/风险/删除」语义。
   - 三色**绝不**互相串用，绝不用于普通强调、普通成功弹窗或普通按钮底色（主色按钮用青绿蓝渐变，不占用状态色）。
6. **AI 卡必标「诊疗辅助建议」**：所有 AI 诊疗辅助卡片顶部标识行必须含「诊疗辅助建议」标题，明确其为辅助建议而非诊断结论；依据蓝条/风险红条/建议绿条结构保留。
7. **禁「等待医生确认」作按钮文案**：医生端操作方是医生本人，按钮用「确认 / 通过 / 驳回 / 接诊」，不用患者视角文案。
10. **登录须用手机号、禁用工号**：医生端登录页使用「手机号输入 + 已绑定医生展示」形态，禁止出现「工号」字段（旧令牌文档 v1.0 的工号写法已作废）。
11. **工作台布局（v2 第二轮整改口径）**：登录进入工作台后，**不再设「诊间快捷入口」图标 grid**（已删除）；「待办」区**仅显示 IM 消息流**（患者发来的消息，如补充资料、咨询用药），禁止把任务型待办（去确认档案/接诊）混入待办区；点 IM 消息可进入对应患者档案。
12. **标准 9 项字段名全屏一致**：详情档案、医生确认、语音接诊抽取回填、看诊前资料摘要等所有屏展示 9 项病史字段时，字段名必须与 §7.2 标准命名完全一致，禁止口语化/简化写法（如「首次发作/部位」「近 1 年发作」「降尿酸用药」「当前疼痛红肿」「长期用药」）。
13. **患者队列「待接诊」单列 + 主线入口闭环**：患者队列顶部须用分段 Tab 将「待接诊」患者单独拉出一列（高亮置顶、橙状态、角标计数），与「全部患者」区分；点任一待接诊患者行 → 进入**该患者的接诊流程页**（复用详情档案屏），屏内顶部须有一排「该患者操作入口」（语音接诊 / 出示就诊码 / 确认纳管），顺次可达各接诊环节，禁止只靠全局导航才能进入语音接诊、就诊码等环节。
14. **常驻「出示就诊码」入口**：工作台顶栏与患者队列顶部均须有常驻「出示就诊码」入口（点击弹就诊码浮层，不离开当前上下文，可带当前患者姓名）；禁止仅在就诊码屏或全局导航里才能出码。
15. **患者接诊流程页 = 信息聚合页**：档案页（screen-profile）须把信息直接铺开，不靠跳转——顶部「智能诊疗」渐变大卡片（明显入口，写清「智能诊疗 · 查看辅助建议」）→ 操作入口排（仅 语音接诊/出示就诊码/确认纳管 三个动作）→ 基本信息 → **看诊前资料摘要（内嵌，不单独成跳转环节）** → 痛风结构化档案(9项) → 化验单归档。禁止让患者已在看某患者时还要再点「看诊前资料」才看到资料。
16. **语音接诊须直进采集界面**：从档案页/工作台点「语音接诊」必须**直接进入并开始录音采集**（自动开始录音动画+转写），禁止弹二次确认或大卡片提示后再让用户点一次才进。
17. **智能诊疗入口须显眼**：档案页顶部「智能诊疗」须为渐变大卡片 + AI 标识 + 明确文案，一眼可知点此进入智能诊疗；禁止把 AI 辅助入口藏在普通小按钮里。
18. **就诊码用真实图片 + 自动刷新**：就诊码屏与就诊码浮层须使用真实二维码图片（`../assets/doctor-qr.png`，相对 demos/ 上一级），禁用 JS 伪 QR；并定时（演示用 15s）自动刷新（重设 img src 加时间戳防缓存），保留 onerror 回退内嵌 SVG。

### 2.3 联动红线（来自 q-0，保留 v1）
8. **双向联动不可断**：两套 Demo 同一运行域（同一 HTML/JS 上下文），患者端建档 → 医生端待确认数实时 +1 → 医生确认 → 患者端即时进入「已纳管」。任意一侧状态变更必须驱动对侧刷新。
9. **重置按钮可见且彻底**：医生端右侧固定可见「重置演示」（q-3），点击清空本轮确认/补录/展开/联动状态，回到工作台初始态，并与患者端重置联动。

## 三、外壳对齐方案（Showcase Shell Alignment）

### 3.1 复用结论
医生端**复用患者端外壳四层结构**，内部视觉同调（两端都用青绿蓝体系），实现「同形同调」。

患者端模板 `iphone-showcase-patient.html` 已提供外壳四层，其类名可直接复用（已在 `component-gallery.html` 中验证）：

| 外壳层 | 复用类名 | 医生端处理方式 |
|-------|---------|---------------|
| 浅色展示背景 | `.ds-showcase-stage` + `.ds-showcase-shadow` | 直接复用（共享外壳，不改色） |
| 模拟器控制条 | `.ds-simulator-bar` / `.ds-simulator-lights` / `.ds-simulator-name` / `.ds-simulator-actions` | 直接复用 |
| iPhone 机身 | `.ds-showcase-device` | 直接复用（`--showcase-device-frame` 黑框 + `--shadow-showcase-device` 投影） |
| 屏幕玻璃反光 | `.ds-showcase-reflection` | 直接复用 |
| **手机内屏** | `.ds-showcase-screen` 内部 | 内屏背景用患者端 `--patient-bg` 渐变或浅色，内部改用医生端 `ds-doctor-*` 组件 |

### 3.2 内屏背景衔接（v2 调整）
v1 用 `--doctor-bg` 灰底区分两端；v2 改为**两端同调**，医生端内屏背景复用患者端浅青绿到浅蓝渐变 `--patient-bg`，不再用灰底制造区分。

```css
/* 医生端内屏（v2）：背景复用患者端 --patient-bg 渐变，与外壳同调 */
.ds-doctor-screen {
  width: 100%;
  height: 100%;
  background: var(--patient-bg);   /* linear-gradient(180deg,#e8f4ff,#f0fff9 52%,#f5f7fb) */
  overflow: hidden;
  position: relative;
  display: flex;
  flex-direction: column;
}
```

### 3.3 外壳 CSS 变量衔接建议
`tokens.css` 已含 `--showcase-*` 与 `--patient-*` 变量，医生端外壳与内屏直接引用，**不新定义外壳/主色变量**。状态栏沿用 `.ds-doctor-statusbar`（42px，深文字）。

> 衔接要点：外壳层（`.ds-showcase-*`）与状态栏是「共享展示容器」，内屏业务区（`.ds-doctor-screen` 及其内部 `ds-doctor-*` 组件）是「医生端专属（高密度版）」。筑原型先抄外壳模板，内屏根节点用 `ds-doctor-screen`（背景 `--patient-bg`），内部组件按 v2 青绿蓝基调实现。

### 4.3 新增屏组件规范草案（v2 大圆角版，参考令牌文档 4.x 写法）

#### 4.11 看诊前资料 · OCR 采集模式卡（新增，合并进屏 #5）
**用途**：医生点击「补充资料 / OCR 识别」时，展示资料采集状态（上传中→识别中→提取完成）与回填结果预览。
**结构**：
```
[资料缩略图] [状态行：识别中…]
[提取结果列表：字段名(右值) + 来源badge「AI识别」]
[操作：确认入库 / 人工修正]
```
```css
.ds-doctor-ocr-card {
  border-radius: var(--radius-xl);          /* v2 大圆角 22px */
  background: var(--doctor-panel, #fff);
  border: 1px solid var(--patient-primary-2);
  padding: 16px;
}
.ds-doctor-ocr-status {
  display: inline-flex; align-items: center; gap: 6px;
  font-size: var(--text-sm); font-weight: 800;
```

### 7.1 内置示例患者字段约定

| 最近时间 | 建档/复查时间 | 2026-07-10 | 复查逾期 21 天 |

### 7.2 痛风结构化档案（9 项病史字段 + 化验单归档）
9 项固定字段（与患者端问诊一致，医生端全量展示）：
1. 首次痛风样疼痛时间
2. 首次发作部位
3. 是否有痛风石
4. 合并症/并发症
5. 当前疼痛和红肿活动状态
6. 长期降尿酸用药情况
7. 当前消炎止痛用药情况
8. 近 1 年发作次数
9. 图片或资料补充

**来源标记挂接规则**（每个字段值后方挂一个来源 badge，见 §5.2(d)）：
- 患者自填 → `患者自填`
- 患者上传（图片/报告）→ `患者上传`
- 录音识别（语音接诊抽取）→ `录音识别`
- AI识别（AI 问诊/依据抽取）→ `AI识别`
- 医生补录（诊间医生手动补）→ `医生补录`
- 医生确认（最终确认写入）→ `医生确认`

> **标准字段口径（强制红线）**：9 项病史字段的**字段名在所有屏必须完全一致**，包括详情档案、医生确认屏、语音接诊抽取回填、看诊前资料摘要等任何展示场景。标准字段名如下（禁止口语化/简化写法，如「首次发作/部位」「近 1 年发作」「降尿酸用药」「当前疼痛红肿」「长期用药」等）：
> 1. 首次痛风样疼痛时间
> 2. 首次发作部位
> 3. 是否有痛风石
> 4. 合并症 / 并发症
> 5. 当前疼痛和红肿活动状态
> 6. 长期降尿酸用药情况
> 7. 当前消炎止痛用药情况
> 8. 近 1 年发作次数
> 9. 图片或资料补充
>
> 注：化验单（血尿酸/肌酐/肝功能 ALT 等）属「化验单归档」维度，与 9 项病史字段分属不同维度，字段名按需展示，不强制套用上述 9 项命名。

**化验单归档**：按报告/化验时间归档为独立分块，每条含：报告时间、血尿酸、肌酐、肝功能、来源标记（患者上传 / OCR识别 / 医生补录）。OCR 识别的化验单在「看诊前资料」OCR 模式下展示识别态。

### 7.3 手机号脱敏强制
所有展示场景手机号统一 `1XX****XXXX` 脱敏格式，**禁止**明文手机号出现在任何屏。

## 八、给筑原型的开工提示（Handoff Notes · v2）

按以下顺序开工，可直接移交：

1. **先抄外壳模板**：从 `design-system/templates/iphone-showcase-patient.html` 复制，保留 `.ds-showcase-*` 四层外壳与 `.ds-simulator-*`，内屏根节点用 `ds-doctor-screen`，背景设 `--patient-bg` 渐变（与外壳同调）。状态栏用 `.ds-doctor-statusbar`。
2. **复用患者端视觉语言**：主色/渐变/圆角/字号/行高/装饰直接套 tokens.css 的 `--patient-*` 体系——顶部栏、主按钮、关键强调用 `--patient-gradient`；卡片大圆角（18–26px）、按钮圆角 20–25px（可用 `--radius-pill`）；行高宽松 1.55–1.6；允许浅青绿到浅蓝渐变背景、轻阴影、轻描边、柔光装饰。
3. **保留医生端高密度组件与操作条**：列表/分块/来源标记密度高于患者端；底部统一用 `.ds-doctor-actionbar`（不用患者端四胶囊）；AI 卡必标「诊疗辅助建议」（依据蓝条/风险红条/建议绿条）；图标用线性 SVG outline（比患者端圆角彩色方块更克制专业）。状态语义色用 `--doctor-orange/green/red` 严格区分。
4. **接 JS 多屏切换**：用 `.ds-doctor-screen` + `.active` 切换，右侧工作流导航点击切屏。
5. **接双向联动**：医生端 JS 暴露状态接口（如 `window.demoState`），患者端同域调用；待确认角标、患者队列、就诊码到院预警随 `demoState` 变化刷新。监听患者端建档/扫码事件 → 驱动医生端角标 +1、列表新增。
6. **接重置演示**：右侧固定「重置演示」按钮 → `resetDemo()` 清空 `demoState`、展开态、补录态、联动态，回到工作台初始；与患者端重置同步。
7. **素材落地**：建档码/就诊码二维码用 `assets/doctor-qr.png` 或内嵌 base64，**禁止**外链临时二维码（规范 §八a）。
8. **红线自检**：交付前逐条核对 §二 红线清单 9 条，尤其是主色为青绿蓝渐变（非旧蓝）、保留医生端高密度/操作条/来源六类/AI 标注/状态色铁律、图标非 emoji/单字、联动不可断、重置彻底。
