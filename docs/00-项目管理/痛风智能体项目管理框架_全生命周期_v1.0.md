# 痛风智能体 · 项目管理框架（全生命周期 v1.0）

> 适用范围：痛风智能体一期（快速建档、医患关联、录音提取、OCR 补充、结构化档案、AI 依据、复诊/失访闭环、患者获得感）。
> 本框架不替代业务文档，而是把 `docs/01-产品需求`、`docs/02-字段字典`、`docs/03-原型与规范`、`design-system`、`demos`、复盘目录串成一套可执行的推进机制。
> 维护方式：本文件每周滚动更新一次；风险与待确认项见 `风险与待确认登记册.md`；排期与任务拆解见 `任务分解与排期模板.md`。

---

## 0. 先说清楚：这个项目用什么节奏管

痛风智能体目前是"强产品驱动 + 专家（主任）决策 + AI 辅助执行"的结构。它不像纯研发项目有固定迭代日历，也不像纯咨询项目只有交付物。它真正的风险不在"代码写不完"，而在三件事：

1. **主任确认链断裂**——两个"待与主任确认"事项（病种选择口径、候诊自填主路径）直接卡住建档表单和病史补充交互的设计，不确认就无法收口。
2. **Demo 与规则漂移**——AGENTS.md 里写死的固定规则（静默登录、手机号补在哪、扫码就诊口径、录音不外露等）一旦在改页面时被无意破坏，对外展示就会露馅。
3. **资产散落**——需求、字段、规范、设计系统已分层，但缺一个"谁在跟、跟到哪、下周交付什么"的单一事实源。

所以本框架的核心不是 PMP 术语，而是三句话：
**确认项先挂号、改动先复述、每周一次收口。**

## 1. 项目启动阶段（Start-up）

### 1.1 目标界定（必须写进一句话北极星）
- **北极星目标（建议口径）**：让医生在 5 分钟内完成痛风专病建档与医患关联，患者候诊即补资料、扫码即就诊，复诊/失访闭环可被医生主动管理。
- 执行标准：北极星必须能被"一期主线 8 个能力"逐条映射，不能出现主线里有、目标里没有的悬空能力。
- 交付物：`docs/01-产品需求/痛风智能体业务流程图定稿.md` 已是目标载体，启动阶段只需补一页《一期范围声明》（见 1.3）。

### 1.2 范围界定（明确"做什么 / 不做什么"）
**一期 IN（已锁定主线）：**
快速建档、医患关联、录音提取、OCR 补充、结构化档案、AI 依据、复诊/失访闭环、患者获得感。

**一期 OUT（默认不做，需主任特批才进）：**
- 多病种通用建档平台（当前只做痛风专病）
- 诊间医生端录音回填为主路径（仅作待确认项，见风险登记册 R-02）
- 患者自选病种（见风险登记册 R-01）
- 医保/处方/药品配送等超出本期闭环的外部系统对接

执行标准：任何新需求进来，先判断落在 IN 还是 OUT。落在 OUT 的，登记到"范围变更请求"而非直接开工。

### 1.3 启动交付物清单（Gate 1 出口标准）
| 交付物 | 责任人 | 完成判据 |
|---|---|---|
| 《一期范围声明》 | 产品（鱼宝/小鹿） | 列出 IN/OUT，主任口头或书面认可 |
| 角色与决策链说明 | 产品 | 明确主任为唯一业务决策人，AI 为执行层 |
| 现有资产盘点表 | 小鹿 | 即本框架引用的 `资产索引.md` 已具备 |
| 风险与待确认登记册 v0.1 | 小鹿 | 见 `风险与待确认登记册.md` |

> Gate 1 不通过，不进入规划排期。

## 2. 项目规划阶段（Planning）

### 2.1 任务分解（WBS）策略
按"能力域 → 页面/模块 → 任务卡片"三级拆。一级域直接对应一期主线 8 项；二级是两端（患者端/医生端）的具体页面；三级是可验收的任务卡。

示例（建档域节选）：
```
1 快速建档
  1.1 患者端：扫码静默登录 + 手机号补充（微信授权/手动）
  1.2 患者端：姓名/性别/手机号三字段表单
  1.3 医生端：工作台外层建档入口 + 痛风专病建档码出示
  1.4 医患关联：扫码建档后自动带出病种并绑定医生
```

执行标准：
- 每张任务卡必须带「验收标准」字段，否则不算拆完。
- 涉及 AGENTS.md 固定规则的卡片，必须在卡内标注"受规则 X 约束"，改动前先复述。
- 完整 WBS 见 `任务分解与排期模板.md`。

### 2.2 排期策略
- **不排固定 sprint 日历**，改用"确认锚点 + 双周滚动"：以"主任确认两个待确认项"为第一锚点，锚点前只做不依赖确认的事（设计系统复用、字段字典补全、OCR 归档逻辑），锚点后放开建档表单与病史补充交互。
- 排期单位用"周"，每张卡估"人日（AI 执行日）"，不估工时。
- 关键路径：建档表单 + 医患关联 + 结构化档案展示，三者串行依赖最强。

执行标准：每周一产出当周《周计划》（最多 5 张高优卡），周五做收口（见第 5 节）。

### 2.3 规划交付物（Gate 2 出口标准）
| 交付物 | 完成判据 |
|---|---|
| WBS 一级~三级拆解 | 8 个能力域全覆盖，每张卡有验收标准 |
| 双周滚动排期表 | 标出确认锚点与关键路径 |
| 沟通机制约定 | 见第 3 节，已写入本框架 |

## 3. 团队协作与沟通机制（Execution 支撑）

### 3.1 角色与分工
| 角色 | 实际承担 | 决策权限 |
|---|---|---|
| 业务决策人（主任） | 真实门诊场景判断 | 唯一可确认"待确认项"的人 |
| 产品负责人（鱼宝） | 需求口径、范围、验收 | 可批准 IN 域内任务开工 |
| AI 执行层（小鹿） | 文档、Demo、字段、规范、排期跟踪 | 边界明确小任务可直接执行；主流程改动先复述 |
| 设计系统 | `design-system/` 复用 | 视觉令牌变更需产品确认 |

### 3.2 沟通节奏（固定动作，不靠临时拉群）
- **每日**：小鹿自检 `demos/` 是否违反 AGENTS.md 固定规则（静默登录、手机号位置、扫码就诊口径、录音不外显），发现问题即时在当前对话提醒。
- **每周五**：项目收口会（可与主任确认合并），输出《周收口记录》（进度 %、本周完成、下周计划、新增风险）。
- **确认锚点触发时**：主任确认任一项 → 当日内更新风险登记册状态 → 解锁对应任务卡 → 复述需求后再改文件。

### 3.3 协作硬规则（来自 AGENTS.md，复述即执行门槛）
任何以下改动，**必须先一小段话复述需求，等确认再动文件**：
- 主流程、页面信息架构变更
- 医生端/患者端关键状态变更
- 主任待确认事项相关
- 对外展示 Demo 修改

边界明确的小任务（改错别字、删内部备注、按规则自检、整理文档）可直接执行并回报。

## 4. 风险识别与应对预案（Risk）

> 完整登记册见 `风险与待确认登记册.md`。此处给方法。

### 4.1 风险分类（本项目高频四类）
1. **决策风险**：待主任确认项悬而未决，卡住设计。
2. **合规/口径风险**：Demo 违反固定规则，对外露馅。
3. **资产风险**：文档/字段/规范版本不一致，AI 读到旧规则。
4. **范围风险**：新需求无 gate 直接进，主线膨胀。

### 4.2 风险登记表字段标准
每条风险必须有：`编号 / 描述 / 类别 / 概率(高/中/低) / 影响(高/中/低) / _owner / 应对预案 / 状态(开放/已缓解/已关闭) / 关联任务卡`。

### 4.3 应对预案模板
- **决策风险** → 挂号到登记册，设确认锚点，锚点前只做不依赖项；准备 A/B 两套原型备用。
- **口径风险** → AGENTS.md 固定规则转为"自检清单"，每次改 Demo 后小鹿跑一遍。
- **资产风险** → 以 `资产索引.md` 为单一事实源，文档改名带版本日期，旧版进 `_archive`。
- **范围风险** → 所有新需求先落"范围变更请求"，Gate 评审后才进 WBS。

## 5. 进度跟踪与质量把控（Monitoring & Control）

### 5.1 进度跟踪方法
- **单一进度板**：以 `任务分解与排期模板.md` 为唯一进度源，不用多套表。
- **完成度口径**：卡片状态只用 `未开始 / 进行中 / 待验收 / 已完成`；"已完成"必须经产品按验收标准确认。
- **周报结构**（固定，见复盘目录风格）：
  ```
  本周完成：__（对照周计划逐条）
  进度%：__（按已完成卡 / 总卡）
  偏差：__（延迟卡及原因）
  下周计划：__
  新增风险：__
  ```

### 5.2 质量把控（QG，Quality Gate）
本项目质量不是"代码跑通"，而是"对外展示不露馅 + 规则一致 + 资产可追溯"。设三道 QG：

| QG | 检查点 | 判据 | 不通过后果 |
|---|---|---|---|
| QG-需求 | 任务卡开工前 | 卡含验收标准、标注受约束规则 | 退回规划 |
| QG-Demo | Demo 改动后 | 不违反 AGENTS.md 固定规则；页面无内部备注 | 小鹿打回重改 |
| QG-文档 | 文档产出后 | 带版本日期、进正确目录、旧版归档 | 不入资产索引 |

### 5.3 一致性自检清单（每次改完 Demo 必跑）
- [ ] 患者端静默登录，手机号补在登录/扫码后页面，不在医生端
- [ ] 快速建档仅姓名/性别/手机号
- [ ] 医生端外层不显眼展示"问诊录音"
- [ ] 痛风病史按结构化分块+字段+值+来源展示
- [ ] 诊前任务串联：补充病史→上传报告→扫码就诊，每步可跳过
- [ ] 无"等待医生确认"主按钮，无"扫码签到"文案
- [ ] 页面无 emoji/符号占位图标，无开发备注

## 6. 项目收尾与经验沉淀（Close-out）

### 6.1 收尾交付物（Gate 3 出口标准）
| 交付物 | 判据 |
|---|---|
| 一期功能验收清单 | 8 能力域逐条验收通过 |
| Demo 最终版 | 通过 QG-Demo，可对外展示 |
| 字段字典终版 | `docs/02-字段字典/` 无待补字段 |
| 规则文档终版 | AGENTS.md + 设计规范无冲突 |
| 复盘报告 | 见 6.2 |

### 6.2 复盘方法（固定四问，落到 `docs/复盘/`）
1. **做对了什么**：本期有效动作（如双周滚动、确认锚点、自检清单）。
2. **踩了什么坑**：规则漂移、确认链延迟等真实事件。
3. **哪个决策最关键**：主任确认的某一项如何解锁了主线。
4. **下次怎么改**：可固化的流程/模板/清单。

执行标准：复盘不写感想，写"可复用动作"；结论回灌本框架或 `任务分解与排期模板.md`。

### 6.3 经验沉淀落到资产
- 固定规则 → 回写 AGENTS.md / 设计规范。
- 排期节奏 → 回写排期模板。
- 风险预案 → 回写登记册。
- 复盘原文 → `docs/复盘/` 带日期归档。
