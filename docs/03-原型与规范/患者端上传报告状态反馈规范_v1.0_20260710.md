## 一、为什么要做页面级状态反馈（需求背景）

### 1.1 问题

原「上传化验报告 / 上传资料」流程只靠一句"上传成功"toast 提示，存在三个体验断点：

- **过程不可见**：上传中、审核中没有任何页面级反馈，患者不知道系统有没有收到、在干什么。
- **失败无出口**：上传失败、审核失败只弹一下 toast 就消失了，患者不知道怎么补救，只能退出重来。
- **重复无拦截**：同一份报告反复上传没有提示，浪费患者时间，也增加医生端去重负担。

### 1.2 目标

把"上传报告"从一个瞬时动作，变成一条**有状态、可回溯、可补救**的闭环流程，让患者在每一步都明确知道：

1. 当前到了哪一步（上传前 / 上传中 / 待审核 / 已完成 / 失败）；
2. 这一步要我做什么（拍照、等待、重新上传）；
3. 出错了怎么办（重试 / 换来源 / 联系医生）。

### 1.3 验收标准

| 验收项 | 标准 |
| --- | --- |
| 状态覆盖 | 覆盖上传前、上传中、上传成功、上传失败、重复上传、审核中、审核通过、审核失败、重新上传共 9 个状态（其中审核中 / 通过 / 失败一期为模拟同步，不计入上线验收硬性项，见第八章） |
| 反馈形式 | 每个状态都有**页面级状态卡**（非仅 toast），关键状态辅以轻提示 |
| 失败可补救 | 上传失败 / 审核失败必须提供"重新上传"或"联系医生"出口 |
| 重复拦截 | 检测到重复报告时先提示，不直接覆盖或静默忽略 |
| 不破坏主流程 | 状态卡出现时不遮挡底部导航，不阻断"跳过 / 继续就诊准备"等主路径 |

## 二、整体设计原则（交互规范基线）

| 原则 | 固定规则 |
| --- | --- |
| 温和医疗级反馈 | 采用 Calm Clinical 风格：状态卡用低饱和色、明确图标、一句人话，不制造焦虑 |
| 页面级优先于 toast | 凡涉及"过程状态"和"结果状态"，一律用上传屏内的状态卡呈现；toast 仅作辅助轻提示 |
| 单一主状态 | 同一时刻上传屏内只显示一个主状态卡，状态切换时整体替换，不堆叠 |
| 操作可逆 | 上传中可取消；上传失败 / 审核失败可重新上传；任何状态都保留返回入口 |
| 来源无关 | 拍照、相册、文件三种来源进入后共用同一套状态机，仅在"上传前"展示对应来源提示 |
| 不新增主题色 | 状态区分靠图标与文案而非新颜色；常规态沿用患者端主青绿（`--patient-primary #03D49F`）/ 主蓝（`--patient-primary-2 #01A1FF`），警告态临时借用医生端 `--doctor-orange (#ED7B2F)`，待设计 token 固化后补患者端 `--patient-warning` 变量并对齐（注：项目目前无正式的"四档语义色"文档，本条以现有 token 为准） |

## 三、九状态总览

> 状态机由 `renderLabUploadState(state)` 驱动，state 取值如下。每个状态对应上传屏 `#labUploadStatusCard` 内的一张状态卡。

| 编号 | 状态 | 触发时机 | 页面级反馈 | 患者可操作 |
| --- | --- | --- | --- | --- |
| 1 | 上传前（idle） | 进入上传屏，未选来源 | 状态卡：标题"上传化验报告"+ 说明"可拍照或从相册选择"，展示来源入口 | 选择拍照 / 相册 / 文件 |
| 2 | 上传中（uploading） | 选来源后开始上传 | 状态卡：加载动画 + "正在上传报告…" + 进度百分比（模拟） | 取消上传 |
| 3 | 上传成功（uploaded） | 文件传到服务端 | 状态卡：成功图标 + "报告已提交，等待医生确认" | 查看结果 / 继续补充 |
| 4 | 上传失败（upload-failed） | 网络或解析异常 | 状态卡：警告图标 + "上传失败，请重试" + 失败原因 | 重新上传 / 换来源 |
| 5 | 重复上传（duplicate） | 检测到同源同名报告 | 状态卡：提示图标 + "这份报告已上传过" + 上次上传时间 | 仍要上传 / 取消 |
| 6 | 审核中（reviewing） | 医生端进入审核 | 状态卡：时钟图标 + "医生审核中" + 预计时长 | 等待 / 联系医生 |
| 7 | 审核通过（approved） | 医生确认报告有效 | 状态卡：成功图标 + "报告已通过，已纳入档案" | 查看档案 / 继续 |
| 8 | 审核失败（rejected） | 医生退回或报告不可用 | 状态卡：警告图标 + "报告未通过" + 退回原因 | 重新上传 / 联系医生 |
| 9 | 重新上传（re-upload） | 失败后用户主动重传 | 回到状态 1/2，状态卡切换为上传前 / 上传中 | 同状态 1、2 |

## 四、各状态交互细节

### 状态 1 · 上传前（idle）

- **触发**：`labUploadIdle()` 或首次进入上传屏。
- **页面呈现**：
  - 顶部状态条：`#labUploadTopStatus` 显示"可补充 · 就诊前可跳过"。
  - 状态卡标题"上传化验报告"，副文案"可拍照或从相册选择，资料越全医生判断越准"。
  - 三个来源入口：拍照上传报告 / 从相册选择 / 从文件选择。
- **患者操作**：点任一来源 → 进入状态 2。
- **注意**：来源入口点击统一走 `selectAttachmentSource(source, label)`，再由 `startReportUploadFlow` 接管。

### 状态 2 · 上传中（uploading）

- **触发**：`driveReportUpload(source, 'upload')` 启动定时器模拟上传。
- **页面呈现**：
  - 状态卡显示环形 / 线性加载 + "正在上传报告…"。
  - 展示模拟进度（0→100%，约 450ms 完成）。
  - 右上角"取消"按钮（`labUploadCancel`）。
- **患者操作**：
  - 等待 → 自动进入状态 3（成功）或状态 4（失败，按模拟结果）。
  - 取消 → 回到状态 1。

### 状态 3 · 上传成功（uploaded）

- **触发**：上传定时器完成且无异常。
- **页面呈现**：
  - 成功图标（✓）+ "报告已提交，等待医生确认"。
  - 展示报告缩略 / 文件名（如有）。
- **患者操作**：
  - "查看结果" → 进入状态 6（审核中）或其结果态。
  - "继续补充资料" / "继续就诊准备" → `goPatient('screen-patient-waiting')`（候诊屏）。

### 状态 4 · 上传失败（upload-failed）

- **触发**：上传定时器标记异常（Demo 中按模拟分支或真实网络错误）。
- **页面呈现**：
  - 警告橙图标 + "上传失败，请重试"。
  - 展示失败原因（如"网络中断""文件无法识别"）。
- **患者操作**：
  - "重新上传" → 状态 9 → 状态 2。
  - "换来源上传" → 状态 1。

### 状态 5 · 重复上传（duplicate）

- **触发**：`prepareLabPhotoUpload` / 来源选择后，`#labRecognizedBlock` 或上传前检测到同源同名报告。
- **页面呈现**：
  - 提示图标 + "这份报告已上传过"。
  - 展示上次上传时间，便于患者判断是否为同一份。
- **患者操作**：
  - "仍要上传" → 状态 2。
  - "取消" → 状态 1。

### 状态 6 · 审核中（reviewing）

- **触发**：医生端进入审核，患者端收到状态推送（Demo 中由"查看结果"或候诊屏状态同步进入）。
- **页面呈现**：
  - 时钟图标 + "医生审核中"。
  - 预计时长提示（如"通常 1 个工作日内"）。
- **患者操作**：等待 / "联系医生"（跳转聊天）。

### 状态 7 · 审核通过（approved）

- **触发**：医生确认报告有效。
- **页面呈现**：
  - 成功图标 + "报告已通过，已纳入档案"。
- **患者操作**："查看档案" → 档案屏；"继续" → 候诊屏。

### 状态 8 · 审核失败（rejected）

- **触发**：医生退回或报告不可用。
- **页面呈现**：
  - 警告橙图标 + "报告未通过"。
  - 展示退回原因（如"报告模糊，请重新拍摄""非本报告期化验单"）。
- **患者操作**：
  - "重新上传" → 状态 9 → 状态 2。
  - "联系医生" → 聊天。

### 状态 9 · 重新上传（re-upload）

- **触发**：状态 4 / 状态 8 中用户主动重传。
- **页面呈现**：复用状态 1（上传前）与状态 2（上传中）的卡片，顶部加一句"正在重新上传"；进入上传中后**完全沿用状态 2 的进度展示与"取消"逻辑**（`driveReportUpload` + `labUploadCancel`），不另起一套。
- **患者操作**：同状态 1、2。

## 五、状态机与函数映射（开发接手参考）

| 函数 | 职责 |
| --- | --- |
| `renderLabUploadState(state)` | 按 state 渲染 `#labUploadStatusCard` 内对应状态卡，整体替换不堆叠 |
| `labUploadIdle()` | 回到上传前状态（状态 1） |
| `openHomeUploadReportDialog()` | 打开上传屏并进入状态 1（等价于 `openAttachmentSheet('report')`） |
| `selectAttachmentSource(source, label)` | 来源选择统一入口；report 类型走 `startReportUploadFlow` |
| `startReportUploadFlow(source, mode)` | 启动上传流程，驱动状态 1→2→3/4 |
| `driveReportUpload(source, mode)` | 模拟上传进度，约 450ms 后落定成功 / 失败 |
| `prepareLabPhotoUpload(...)` | 拍照 / 相册选择后预处理，含重复检测（状态 5） |
| `labUploadCancel()` | 上传中取消，回到状态 1 |
| `viewLabResults()` | 从上传成功态进入审核中 / 结果态（状态 6/7/8） |

**关键容器**

- `#screen-patient-lab-upload`：上传屏。
- `#labUploadTopStatus`：屏顶部状态条（"可补充 · 就诊前可跳过"）。
- `#labUploadStatusCard`：状态卡容器，由 `renderLabUploadState` 整体写入。
- `#labRecognizedBlock`：重复检测 / 识别结果展示区。

## 六、与其他屏的边界

| 关联屏 | 关系 |
| --- | --- |
| 候诊屏 `screen-patient-waiting` | 上传成功 / 审核通过后的"继续就诊准备"按钮 `goPatient` 至此；该屏在整合版中已补回，路径已通 |

| 聊天 | "联系医生"从审核中 / 失败态跳转；聊天发附件走通用附件逻辑，不依赖上传状态机 |
| 档案屏 `screen-patient-myrecord` | "查看档案"从审核通过态跳转，展示已纳入的报告 |

## 七、Anti-Slop 自检（质量门控）

| 检查项 | 结果 |
| --- | --- |
| 九状态是否全部有页面级反馈（非仅 toast） | ✅ |
| 失败态是否都有补救出口 | ✅（重新上传 / 换来源 / 联系医生） |
| 是否引入第四种主题色 | ✅ 未引入，沿用患者端主青绿/主蓝，警告态临时借 doctor-orange |
| 状态卡是否遮挡底部导航 | ✅ 不遮挡，返回入口始终可达 |
| 重复上传是否有拦截而非静默覆盖 | ✅ |

## 八、待与主任确认

- **审核态的真实触发源**：当前 Demo 中审核中 / 审核通过 / 审核失败由"查看结果"或候诊屏同步进入，一期是否需要对接医生端真实回写推送，需在主任确认后补真实接口。
- **重复检测粒度**：Demo 按"同源同名"模拟，真实场景是否需按"报告期 + 化验项目"去重，待医生端确认。
- **失败模拟开关**：Demo 上传失败为模拟分支，上线前需替换为真实网络 / 解析错误捕获。
