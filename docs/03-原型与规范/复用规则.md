# 复用规则

## 基本原则

痛风智能体的患者端和医生端不是两套随机页面，而是同一条专病管理流程的两个角色视角。患者端负责轻量引导、资料补充、复诊提醒和获得感；医生端负责接诊、确认、追溯和闭环。新增页面或改页面时，先判断属于哪个角色、哪个状态、哪个动作，再选择已有组件，不临时发明新样式。

## 患者端

患者端统一使用阿福式陪伴体验。背景使用浅青绿到浅蓝氛围，内容由首页状态卡、微信式 AI 气泡、快捷胶囊和底部输入区承接。除扫码、联系医生、病历夹总览等强任务外，优先在当前对话里完成。

患者端固定复用这些组件：

| 语义 | 组件 |
|---|---|
| 当前管理状态、建档完成反馈、已纳管复诊提醒 | `ds-patient-hero` |
| AI 回复 | `ds-chat-message ds-chat-message-ai` |
| 患者回复 | `ds-chat-message ds-chat-message-user` |
| 推荐问题、快捷问法 | `ds-question-card` |
| 对话内选项 | `ds-chip`、`ds-chip-primary` |
| 底部固定入口 | `ds-shortcut` |
| 输入区 | `ds-compose` |
| 病历夹摘要 | `ds-record-card`、`ds-record-timeline` |

患者端不要把“联系医生、上传报告、健康打卡、饮食问答”在同一页面重复放成多个大入口。底部快捷胶囊已经覆盖这些能力时，内容区只在当前任务确实需要时出现一次任务级操作。

## 真机演示外壳

对外汇报、封面图、会议预览或单流程展示需要“手机里已经跑起来”的感觉时，使用 iPhone 真机演示外壳，不再临时手写一套手机框。

固定调用这些组件：

| 语义 | 组件 |
|---|---|
| 浅色展示画布 | `ds-showcase-stage` |
| 手机底部投影 | `ds-showcase-shadow` |
| iPhone 机身 | `ds-showcase-device` |
| 手机屏幕 | `ds-showcase-screen` |
| 屏幕反光 | `ds-showcase-reflection` |
| 顶部模拟器条 | `ds-simulator-bar`、`ds-simulator-lights`、`ds-simulator-name`、`ds-simulator-actions` |
| 手机内屏首屏 | `ds-mobile-app`、`ds-mobile-hero-card`、`ds-mobile-card`、`ds-mobile-summary`、`ds-mobile-tabs` |

场景模板在 `design-system/templates/iphone-showcase-patient.html`。需要做新的真机展示页时，先复制这个模板，再替换手机内屏文案和卡片内容。不要把已有带手机外框的完整 HTML 直接 iframe 到真机外壳里，否则会出现“手机里套手机”。真机外壳只服务展示，不替代正式业务 Demo；需要完整点击流程时打开正式 Demo 文件。

## 医生端

医生端统一使用低圆角、高密度、任务优先的工作台风格。主线是待接诊、诊间采集、字段确认、写入档案、生成随访计划。医生端外层不要把问诊录音当作显眼模块，录音只作为字段来源、AI 依据和证据材料出现。

医生端固定复用这些组件：

| 语义 | 组件 |
|---|---|
| 医生工作台头部 | `ds-doctor-head` |
| 三个关键数字 | `ds-stat-grid`、`ds-stat` |
| 当前主任务 | `ds-main-task` |
| 待办列表 | `ds-todo-list`、`ds-todo` |
| 患者状态 | `ds-status-chip` |
| 字段确认 | `ds-field-card` |
| 来源和依据 | `ds-source-badge`、`ds-evidence` |

医生端不要把完整档案平铺到第一屏。工作台第一眼应该回答“现在接诊谁、下一步做什么”，字段确认页再展示可确认、可修改、可追溯的结构化字段。

## 文案边界

界面只写患者或医生当下需要看到的状态、操作和业务内容。不要在页面上写产品逻辑、设计意图、演示流程、字段 ID、开发说明或内部备注。需要解释的规则写到 `痛风智能体Demo设计规范_AI执行版.md` 或 `docs/` 文档里。

## 改动前自检

修改主流程、页面信息架构、医生端/患者端关键状态、主任待确认事项或对外展示 Demo 前，先复述要改的角色端、状态、目标和不要出现的内容，等确认后再改文件。修错别字、删除内部备注、整理既有文档、抽取设计组件这类边界明确的小任务可以直接执行。

### Demo 一致性自检脚本（lint-demo）

`design-system/lint-demo.py` 把本规范和《Demo 设计规范_AI执行版》里可机器判定的固定规则转成可重复检查，替代每次改 Demo 都靠人工复述确认。改完任意 Demo 后跑一次：

```bash
python3 design-system/lint-demo.py            # 默认扫 demos/ 与根目录 *.html
python3 design-system/lint-demo.py 某个文件.html  # 只查指定文件
```

当前覆盖 4 类硬规则：① 快速建档字段洁癖（只应含姓名/性别/手机号，档案详情页展示年龄/身份证/病案号属合规豁免）② 禁用词（建档进度/报告解读/AI诊室/扫码签到/等待医生确认主按钮等）③ 图标规范（单字占位图标、语义 emoji 作功能图标）④ 素材稳定性（不引用临时/本机/易失路径）。输出 `[ERROR]`（阻断级，应修）与 `[WARN]`（疑似，需人工确认）。语义级检查（阿福式氛围、同页入口去重、对话节奏）仍需人工。规则随规范迭代，新增可机器判定项时同步扩展脚本。

**lint 已排除的噪声（不再误报）**：`<script>` JS 代码块、含 `ui-icon`/`aria-hidden` 的字体图标、档案详情容器内的年龄/身份证字段。

**图标体系已统一收口至 components.css（2026-07-13 执行）**：全部 31 个 `ui-icon-*` 线性 CSS 图标类（含 `lightbulb`/`robot`）已迁入 `design-system/components.css`，三份 Demo（患者端 V0.1、病历夹tab、医生端降噪版）均删除内联定义并改 `<link>` 引用 `../../design-system/components.css`。新增/修改图标只改 components.css 一处，不再散落各 Demo。emoji→图标替换（⏳/🎤/🤖/💡/✅/✏/📱/♙）同步完成，三份 Demo 经 `lint-demo` 验证 WARN=0 / ERROR=0。注意 components.css 用 `ds-` 前缀（128 类）与 Demo 内联 `ui-`/`afu-` 等业务前缀无命名冲突，link 后样式互不影响。

## 新页面空白骨架（消除重复造样式）

新做患者端或医生端页面时，**先复制骨架再填文案**，不要从零写样式或新发明组件：

| 角色端 | 骨架文件 | 含结构 |
|---|---|---|
| 患者端 | `design-system/templates/blank-patient.html` | ds-patient-hero 状态卡 + 对话气泡示例 + 底部四胶囊 + 输入框；阿福式浅青蓝氛围 |
| 医生端 | `design-system/templates/blank-doctor.html` | ds-doctor-head + 三 stat + 主任务卡 + 待办列表 + 字段确认卡；蓝色工作台 |

两个骨架均引用 `../components.css`，已通过 `lint-demo` 硬规则检查，可作为合格起点。复制后只改文案、状态、列表项；需要新组件时先回 `component-gallery.html` 看是否已有同语义组件可复用。
