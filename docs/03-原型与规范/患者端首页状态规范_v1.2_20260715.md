# 患者端首页状态规范 v1.2

> 版本：v1.2　日期：2026-07-15　适用端：痛风智能体患者端小程序

## 一、首页整体原则

| 原则 | 固定规则 |
| --- | --- |
| 首页核心目标 | 用一个稳定的首页状态区告诉患者当前管理状态，并给出下一步入口 |
| 卡片风格 | S3/S4/S5/S6-A/S6-B 统一使用同一套已纳管状态卡，只替换业务内容 |
| 底部快捷入口 | 始终保持「联系医生 / 上传报告 / 健康打卡 / 饮食问答」四个按钮，不因状态变化而增减、挪动或改样式 |
| 病历夹入口 | S2-S6 统一展示「我的病历夹」，放在中间信息卡右上角（源文件原位置），不放到底部快捷入口中 |
| 机器人消息 | 只说一句对患者有帮助的下一步建议，不重复卡片中的日期、倒计时、医生姓名、检查项目等结构化信息 |
| 无数据处理 | 没有数据的字段整行隐藏，不展示空值、横线或占位符 |
| 扫码后承接 | 不新增「到诊」或「正在就诊」首页状态，也不进入独立到诊页、就诊中页；扫码成功后返回扫码前首页，仅在首页对话区追加一条结果消息 |

## 二、首页页面区域定义

| 区域 | 页面位置 | 包含内容 |
| --- | --- | --- |
| **上方状态栏** | 页面顶部 hero 区（问候语下方、内容卡片上方） | 问候语「你好，{name}」、状态大标题（随状态变化）、状态说明（标题下小字）、右侧吉祥物 |
| **中间信息卡** | 上方状态栏下方的统一卡片模块 | 「我的病历夹」入口（右上角）、弱化状态说明、数据行（下次复诊 / 倒计时 / 复查项目等）；**不含进度条**（2026-07-10 设计决策删除） |
| **主行动 · 次行动** | 中间信息卡下方 | 主行动按钮（视觉权重高，如「补充复诊资料」「联系医生」）+ 次行动入口（如扫码就诊链接、联系医生） |
| **下方机器人消息** | 主行动下方、底部快捷胶囊上方 | 一句固定引导文案；S6-A 默认隐藏 |

上方状态栏的状态标题即为当前状态的概括（如"待复诊"）。S1 未建档时上方状态栏不展示标题与说明，仅保留问候语与吉祥物。

## 三、首页状态判断顺序

| 优先级 | 首页状态 | 业务含义 | 命中条件 |
| --- | --- | --- | --- |
| 1 | S5 逾期未复诊 | 患者已经超过建议复诊时间，但本次复诊还没有完成 | 已纳管，复诊日期已过，且本次复诊未完成 |
| 2 | S4 复诊当天 | 今天就是医生建议复诊的日期 | 已纳管，当前日期等于建议复诊日期 |
| 3 | S2 建档中 | 患者已提交基本信息，医生还没有完成确认 | 已提交快速建档，医生尚未确认回写档案 |
| 4 | S3 待复诊 | 医生已经安排未来复诊 | 已纳管，有未来复诊计划 |
| 5 | S6-A 确认无复诊需求 | 医生明确判断当前无需复诊 | 已纳管，医生明确标记无需复诊 |
| 6 | S6-B 暂无复诊安排 | 系统里暂时没有医生下发的复诊计划 | 已纳管，但无复诊计划，也无无需复诊标记 |
| 7 | S1 未建档 | 患者还没有进入痛风专病管理 | 无痛风专病档案，或未提交快速建档 |

患者完成扫码后，首页仍按本表 S1-S6-B 的既有判断顺序展示，不因本次已关联接诊而切换为新的首页状态。医生端开始接诊、当天结束或隔天补录均不触发患者端跳页。医生正式保存记录后，系统发送订阅消息；患者进入小程序时拉取最新首页和病历夹数据，并按既有规则更新为 S3、S6-A 或 S6-B。

## 七·A、档案完整度定义

档案完整度 `{archiveCompletion}%` 仅在 **S2 建档中** 状态展示，表示患者侧诊前资料的完善比例。

- **统计口径**：只计患者自己可补充的资料项，不含「医生待确认」。具体由以下待补资料按**字段数**加权计算：
  - 痛风病史：9 项结构化字段（`gout_history.*`）
  - 化验报告：以「是否上传报告原图」计为 1 个资料项
- **权重规则**：以所有待补资料字段总数为分母，已完成字段数为分子，按比例折算百分比。例如病史 9 项 + 报告 1 项 = 10 个资料项；病史完成 9 项、报告未传 = 90%；病史与报告均完成 = 100%。
- **展示方式**：只显示百分比数值文本，**不绘制进度条**（与第七章「禁止恢复项」一致）。
- **跳过处理**：患者跳过某资料项时，该项不计入已完成，但完整度不为 0；跳过仅影响完整度数值，不阻断扫码就诊。

## 八、中间信息卡视觉规范（2026-07-10 设计决策）

### 8.1 卡片形式
- 圆角白/浅底卡片：`border-radius: 24px`，浅绿→浅蓝渐变底 `linear-gradient(135deg,#F0FFF9 0%,#E8F6FF 100%)`，柔和阴影 `0 10px 24px rgba(1,161,255,0.08)`，同色系细边框 `1px solid rgba(3,212,159,0.18)`
- 不使用左侧硬塞的渐变图标方块；不展示进度条

### 8.2 信息层级（从上到下）
| 层级 | 元素 | 视觉权重 |
| --- | --- | --- |
| 三级（最弱） | 弱化状态说明「已纳入痛风专病管理」 | 12px 灰 700，不作大标题；**仅 S3–S6 展示，S1/S2 不展示** |
| 独立入口 | 「我的病历夹」按钮 | 右上角，不变 |
| 一级（主角） | 数据行（各状态不同，如 S2「档案完整度 / 待补资料」、S3「下次复诊 / 剩余天数 / 复查项目」、S5「原复诊日期 / 已逾期」等） | 数值 15px 深蓝 800，字段名 11px 淡灰 700，每行前带淡蓝小图标容器 |
| 分隔 | 渐变装饰线 | 高 3px、青绿→蓝渐变、半透明，纯视觉分隔 |
| 三级（最弱） | 「已关联 {doctorName} 医生 · 痛风专病」 | 装饰线下方极淡 footer，11px 淡灰；**S1 不展示，S2 起展示** |

## 十、扫码就诊入口专项汇总

> 统一口径：患者端一律称「扫码就诊」或「扫接诊码」；**严禁出现「扫码签到」**；接诊码由医生端出示，患者端只扫不展示。

| 状态 | 是否展示扫码就诊 | 入口文案 | 位置 | 触发动作 |
| --- | --- | --- | --- | --- |
| S1 未建档 | 否 | — | — | 应扫「建档码」进入专病管理，非接诊码 |
| S2 建档中 | 资料完成/跳过时展示 | 「扫码就诊」 | ③主按钮 | 调起扫码，识别接诊码→建立本次接诊关联 |
| S3 待复诊 | 是 | 「提前到院也可扫码就诊」 | ③次行动 | 同上 |
| S4 复诊当天 | 是（必展示） | 「我已到院，扫码就诊」 | ③主按钮 | 同上 |
| S5 逾期未复诊 | 是 | 「已到院？扫码就诊」 | ③次行动 | 同上 |
| S6-A 确认无复诊需求 | 否 | — | — | 医生已判无需复诊 |
| S6-B 暂无复诊安排 | 否 | — | — | 引导联系医生安排，不由患者扫码 |  

### 扫码成功后的返回规则

患者从 S2、S3、S4 或 S5 调起扫码并识别有效接诊码后，系统建立本次接诊关联，随即返回发起扫码前的首页状态，并在首页对话区追加消息：「已完成扫码就诊，资料已同步给医生。医生完成记录后会通知你。」患者端不展示独立「到诊页」或「就诊中页」，也不因医生开始接诊、未操作、门诊结束或隔天补录而跳转页面。

本次接诊未由医生正式保存前，原首页的状态栏、信息卡、按钮和功能入口全部保持不变。患者端不持续轮询接诊状态，仅在小程序启动、从后台回到前台、进入首页或患者下拉刷新时查询一次最新数据。

医生正式保存记录后，系统向患者发送订阅消息：「本次门诊记录已更新，点击查看诊疗记录和复诊安排。」患者点击消息进入小程序，或之后正常进入小程序时，客户端拉取最新首页和病历夹数据，再按本章状态切换规则更新首页。订阅消息未授权或发送失败时，不影响落档；患者下次进入小程序时仍可同步最新数据。
