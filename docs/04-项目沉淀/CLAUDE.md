# CLAUDE.md — 痛风智能体项目规则与上下文记忆

> 本文档为 AI 开发指令，供新对话开启时自动加载。包含项目核心业务逻辑、代码规范、字段规范、开发工作流和避坑指南。**每次修改主流程、页面架构或关键状态前，必须先读本文档。**

## 一、项目上下文

### 1.1 项目本质

这是一个**痛风专病管理小程序**的 Demo 原型项目，不是生产环境应用。核心目标是用可交互的 HTML 原型验证产品流程，面向患者端（微信小程序质感）和医生端（专科工作台风格）。

### 1.2 核心业务逻辑

**一期主线 8 项能力**：快速建档 → 医患关联 → 录音提取 → OCR 补充 → 结构化档案 → AI 依据 → 复诊/失访闭环 → 患者获得感。

**端到端流程**：
```
医生端发起痛风专病建档码 → 患者扫码 → 微信静默登录 → 
手机号补充（患者端优先，医生端兜底）→ 快速建档（姓名/性别/手机号）→ 
补充痛风病史（9项结构化问诊，允许跳过）→ 上传化验报告（允许跳过）→ 
到院扫接诊码 → 医生查看资料 → 医生接诊时直接修正档案 → 
患者端进入已纳管首页 → 复诊管理闭环
```

### 1.3 技术架构

- **前端**：纯 HTML/CSS/JS，单文件 Demo，零依赖构建工具
- **样式**：CSS 自定义属性（`design-system/tokens.css`）
- **组件**：自建设计系统（`design-system/components.css`，前缀 `ds-`）
- **图标**：纯 CSS 代码图标（`.ui-icon`），不用 emoji
- **文档图表**：Mermaid
- **版本管理**：Git

### 1.4 关键角色

| 角色 | 实际承担 | 权限 |
|------|---------|------|
| 业务决策人（主任） | 真实门诊场景判断 | 唯一可确认「待确认事项」 |
| 产品负责人（鱼宝） | 需求口径、范围、验收 | 批准 IN 域内任务 |
| AI 执行层（小鹿） | 文档、Demo、字段、规范 | 边界明确小任务可直接执行 |

## 二、代码规范

### 2.1 文件组织

```
新增页面 → demos/{患者端|医生端}/功能描述.html
修改组件 → design-system/components.css（追加，不改已有类名）
新增令牌 → design-system/tokens.css（追加变量，不改已有值）
新增文档 → docs/{分类}/文档名.md
```

### 2.2 命名风格

| 类别 | 规范 | 示例 |
|------|------|------|
| CSS 组件类名 | `ds-` 前缀 + 语义化 kebab-case | `ds-patient-hero`、`ds-chat-bubble` |
| CSS 工具类 | `ui-` 前缀 | `ui-icon`、`ui-icon-qr` |
| CSS 变量 | `--` 前缀 + 角色前缀 | `--patient-primary`、`--doctor-bg` |
| 患者端页面状态 | `S1` ~ `S6` 编号 | S1 未建档、S2 建档中 |
| Demo 文件命名 | 角色端_功能描述.html | `医生端_信息降噪版Demo.html` |
| 文档文件命名 | 主题_版本号_日期.md | `患者端字段字典_v1.1.1_20260706.md` |

### 2.3 注释要求

- Demo HTML 中的 CSS/JS 可以使用简洁的功能注释
- **禁止**在 HTML 页面中写产品逻辑说明、设计意图、演示流程、字段 ID、占位备注
- 需要解释的规则写入 `docs/` 目录下的 Markdown 文档
- 页面文案只保留用户/医生当下需要看到的内容

### 2.4 禁止引入的新依赖

- **禁止**引入任何 npm 包、CDN 库、外部 CSS 框架
- **禁止**使用 emoji 作为功能图标
- **禁止**使用外部图片链接，所有素材必须放入 `assets/` 并相对路径引用
- **禁止**将设计系统文件（`tokens.css`/`components.css`）复制到单个 Demo 中——应通过 `<link>` 引用

### 2.5 HTML 文件结构模板

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>痛风智能体 · 页面名称</title>
  <link rel="stylesheet" href="../../design-system/tokens.css">
  <link rel="stylesheet" href="../../design-system/components.css">
  <style>
    /* 页面特有样式 */
  </style>
</head>
<body>
  <!-- 页面内容 -->
  <script>
    // 页面逻辑
  </script>
</body>
</html>
```

## 三、字段与数据规范

### 3.1 字段命名约定

所有字段以 `docs/02-字段字典/痛风专病智能体患者端字段字典_v1.1.1_20260706.md` 为唯一权威来源。

| 命名约定 | 示例 |
|---------|------|
| 实体前缀 + 下划线 + 字段名 | `patient_profile.name`、`gout_archive.first_pain_time` |
| 枚举值使用小写下划线 | `archive_status.managed` |
| 状态枚举不直接暴露给患者端 | 页面文案用中文状态，后台用英文枚举 |

### 3.2 必填校验规则

| 字段 | 必填 | 校验规则 |
|------|------|---------|
| 姓名 (`patient.name`) | 是 | 非空 |
| 性别 (`patient.gender`) | 是 | 男/女 |
| 手机号 (`patient.mobile`) | 是 | 11 位数字，前端即时校验 |
| 管理病种 (`relation.disease_code`) | 是 | 由建档码带入，患者不选 |
| 痛风病史 9 项 | 否 | 每一步允许跳过 |
| 化验报告 | 否 | 允许跳过，不阻断扫码就诊 |

### 3.3 类型定义标准

| 类型 | 表示方式 |
|------|---------|
| 文本 | `string` |
| 数字 | `number` / `int` / `float` |
| 日期 | `date`（格式 YYYY-MM-DD） |
| 日期时间 | `datetime`（ISO 8601） |
| 单选 | `enum`（枚举值列表） |
| 多选 | `enum[]`（枚举值数组） |
| 文件 | `file`（引用 attachment_file 实体） |
| 布尔 | `boolean` |

### 3.4 核心状态枚举

```javascript
// 建档状态
const ARCHIVE_STATUS = {
  NONE: 'none',              // 未建档
  BASIC_SUBMITTED: 'basic_submitted',  // 建档中

  MANAGED: 'managed',                 // 已纳管
  LOST: 'lost'                        // 已失访

// 复诊状态
const FOLLOWUP_STATUS = {
  NOT_DUE: 'not_due',        // 未到期
  DUE_TODAY: 'due_today',    // 今日复诊
  ARRIVED: 'arrived',        // 已到院
  WAITING_RECORD: 'waiting_record',  // 等待记录
  COMPLETED: 'completed',    // 已完成
  OVERDUE: 'overdue',        // 逾期
  RESCHEDULED: 'rescheduled', // 已改期
  AT_RISK: 'at_risk'         // 失访风险
```

## 四、开发工作流

### 4.1 新增功能标准步骤

1. **分析**：判断属于患者端还是医生端、哪个状态、哪个动作
2. **查规范**：先读 `docs/03-原型与规范/痛风智能体Demo设计规范_AI执行版.md`
3. **查复用**：先查 `design-system/components.css` 是否有可复用组件
4. **复述确认**（涉及主流程/关键状态/待确认事项时）：
   - 用一小段话复述：要改哪个角色端、哪个状态、希望变成什么、哪些内容不要出现
   - **等待用户确认后再动手**
5. **实现**：在 `demos/` 下创建或修改文件，引用 `design-system/` 资源
6. **自检**：逐项检查是否违反 AGENTS.md 固定规则
7. **更新文档**：如果新增了通用组件或规则，同步更新 `components.css` 或设计规范
8. **Git 提交**：提交并推送

### 4.2 修改 Bug 标准步骤

1. 定位问题页面和状态
2. 确认是 CSS 样式问题还是交互逻辑问题
3. 边界明确的小任务（修错别字、删内部备注、整理文档）→ **直接执行**
4. 涉及主流程的修改 → **先复述再执行**

### 4.3 修改时禁止做的事

- **禁止**改变已有 CSS 类名的语义（如需新样式，追加新类名）
- **禁止**修改 `tokens.css` 中的已有变量值（如需新颜色，追加新变量）
- **禁止**在页面中新增"开发说明""演示流程""占位备注"
- **禁止**引入新的外部依赖

## 五、避坑指南（绝对不能触碰的红线）

### 5.1 文案红线

| 禁用文案 | 替代表达 | 原因 |
|---------|---------|------|
| 「扫码签到」 | 「扫码就诊」「扫接诊码」 | 接诊码不是签到，建立的是就诊关联 |
| 「签到成功」 | 「已完成扫码就诊」 | 同上 |
| 「预约成功」 | 「复诊提醒已记录」「请按医院规则挂号」 | 复诊提醒不是预约 |
| 「AI 已诊断」 | 「已整理为医生可查看的资料」 | AI 不做医疗诊断 |
| 「AI 已确认」 | 「已整理为医生可查看的资料」 | AI 不做医疗诊断 |
| 「OCR 已确认异常」 | 「报告结果仅供参考，以医生诊断为准」 | OCR 结果直接入档，医生可修正 |
| 「已自动写入正式档案」 | 「已更新档案」 | 患者提交直接入档 |
| 「等待医生确认」（作为主按钮） | 「继续补充资料」「查看已提交资料」「返回首页」 | 避免患者被动等待 |

### 5.2 逻辑红线

1. **手机号补充以患者端为优先路径**，但医生端快速建档页也需保留手机号录入字段作为兜底。双端来源需支持去重和合并逻辑
2. **快速建档只能有三个字段**：姓名、性别、手机号。年龄、病史、化验、病种选择都不能放入快速建档页
3. **病种由建档码自动带出**，患者端不展示病种选择字段
4. **患者提交内容直接写入正式档案**，医生可随时在接诊/档案编辑时修正覆盖
5. **建档码和接诊码必须严格区分**，不能混用流程
6. **复诊提醒不能表达为预约结果**，页面必须提示按医院实际规则挂号
7. **医生端外层不能把"问诊录音"作为显眼模块**，录音只作为字段来源和证据材料
8. **痛风病史必须按结构化表单顺序展示分块、字段、填写值和来源**，不能只展示一个总答案

### 5.3 状态流转红线

1. **已纳管患者再次扫描建档码时，不能重复创建档案**
2. **过期/无效的建档码或接诊码，不能创建错误档案或错误关联**
3. **医生可随时修正档案中的任何字段**，不需要独立的"确认"步骤
4. **回写失败时，患者端保持当前状态并支持重试**
5. **患者跳过病史或报告不影响扫码就诊**，跳过只是记录未完成状态

### 5.4 视觉红线

1. **页面不能出现开发说明、字段 ID、内部规则解释、占位备注**
2. **患者端统一阿福式对话容器**，不另起一套视觉风格
3. **不使用 emoji 作为功能图标**，统一用 `.ui-icon` CSS 图形
4. **同一功能入口不在同一页面重复出现**（如底部已有「上传报告」，页面内不再放同功能大按钮）
5. **所有对外展示素材必须放入 `assets/` 并相对路径引用**，禁止外链或临时路径

### 5.5 数据红线

1. **涉及患者手机号、身份证号、报告图片等敏感信息时，Demo 使用模拟数据**
2. **字段来源优先级**：医生端编辑 > 患者提交 > OCR > AI 整理（后者可被前者随时覆盖）
3. **同一字段存在多来源时，以最高优先级来源为准**，医生端编辑可直接覆盖其他来源
4. **字段字典 (`v1.1.1`) 是字段唯一权威来源**，不在 Demo 中另造字段

## 附：跨端说明（患者端 ↔ 医生端）

本文件为**患者端**开发规则。医生端具备患者端不涉及的能力，其开发规则见 [`医生端_CLAUDE.md`](./医生端_CLAUDE.md)，完整逻辑基线见 [`INDEX.md`](./INDEX.md)：

- **医生端可主动建档（无码）**：工作站「+ 建档」填三字段直接建档案，不依赖患者扫任何码（ADR-D09）。
- **医生端可事后拍照补录**：接诊后/跨日均可进入患者档案拍照补拍外院病历，OCR 直接入档，不依赖患者在线（ADR-D09）。
- **弱接诊**：患者扫接诊码即资料可见，医生未点「接诊患者」也能采集/OCR/AI 辅助；未接诊患者安静留在「待接诊」分组，超时触发「到院未接诊」预警（ADR-D01）。

修改任一端的主流程、页面架构或关键状态时，须同步检查对端文档是否需对齐，并以 [`INDEX.md`](./INDEX.md) 第二节逻辑基线为准。
