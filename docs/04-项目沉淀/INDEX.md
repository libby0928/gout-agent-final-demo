## 二、逻辑基线（8 份文档共同遵守）

以下决策贯穿患者端 + 医生端全部 8 份文档，改动任一文档须保持口径一致。

**使用方式（替代人肉逐文件核查）**：改任一文档前，先在下方找到受影响决策，按"必须同步落点"列逐一核对；改完在"最近核对"列更新日期。任一文档与本节冲突时以本节为基准并同步修正该文档。

| 编号 | 决策 | 要点 | 必须同步落点（改此决策需核对这些文件） | 最近核对 |
|------|------|------|------|------|
| ADR-002 / D02 | 手机号双端补充 | 患者端（微信授权/手动）优先，医生端快速建档页保留手机号录入兜底；双端来源按最近有效录入去重合并 | 患者端字段字典 v1.1.1、医生端快速建档页、不同端口资料覆盖原则 | 2026-07-10 |

## 三、端到端闭环（患者端 ↔ 医生端）

```
患者扫建档码 → 患者端快速建档（姓名/性别/手机号）
       │         （医生端也可主动无码建档，ADR-D09）
       ▼
患者端补充病史 / 上传报告（生成候选资料，ADR-D03）
       │
       ▼
患者扫接诊码 → 医生端弱接诊（资料立即可看，ADR-D01）
       │         未点接诊的患者：待接诊分组 / 跨日搜索 / 到院未接诊预警
       ▼
医生端诊间采集（语音→AI 提取候选）/ 事后补拍 OCR（原图保存、候选待确认，ADR-D09）
       │
       ▼
医生端 AI 诊疗辅助 → 医生确认执行治疗建议（ADR-D04 硬门槛）
       │
       ▼
字段回写档案 → 患者端进入已纳管首页（病历夹 / 复诊提醒 / 联系医生）
       │
       ▼
复诊逾期 → 医生端逾期提醒 → 触达患者 → 闭环
```

**患者端无需感知医生端内部操作**（建档、接诊、OCR 补拍、资料确认等），候选资料未确认时仅呈现"已提交"等状态；正式回写成功后再呈现"已更新档案""已纳管"等结果态。医生端保留全部来源标记、原始依据和确认状态用于追溯。

## 四、阅读建议

- **新接手开发者**：先读 `INDEX.md` → 对应端 `PROJECT_SUMMARY.md` → 对应端 `CLAUDE.md` → 打开 `demos/` 预览
- **产品/演示**：先读 `INDEX.md` 第二节逻辑基线 → 患者端读取唯一权威 PRD，医生端读取 `医生端_PRD_决策日志.md`
- **测试同学**：直接读对应端 `TEST_PLAN.md`，按 P0 > P1 > P2 优先级执行
- **跨端对齐**：以本索引第二节"逻辑基线"为准，任一文档与本节冲突时以本节为基准并同步修正该文档
