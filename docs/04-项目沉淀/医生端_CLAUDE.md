# 医生端_CLAUDE.md — 痛风智能体医生端项目规则与上下文记忆

> 本文档为 AI 开发指令，供新对话开启时自动加载。包含医生端核心业务逻辑、代码规范、字段规范、开发工作流和避坑指南。**每次修改医生端主流程、页面架构或关键状态前，必须先读本文档。**

---

## 一、项目上下文

### 1.1 项目本质

这是一个**痛风专病管理小程序**的 Demo 原型项目，医生端是其中的专科工作台视角（蓝色、高密度、任务优先），与患者端（青绿、引导式）共享同一份痛风专病档案。

### 1.2 核心业务逻辑

**一期主线**：医生登录 → 今日工作站 → 多入口建档/关联患者 → 诊间采集（语音/OCR/补录）→ 看诊前资料汇总 → AI 诊疗辅助（医生确认执行）→ AI 病史小结沉淀 → 联系患者 / 逾期提醒闭环。

**重要：接诊是弱动作（弱接诊）**。患者扫接诊码建立就诊关联后，详情页资料、语音采集、OCR、AI 辅助**立即可用**，不强制医生先点「接诊患者」。「接诊患者」仅标记"开始本次看诊"用于计时/排队，不是看资料的门禁。忙时未点接诊的患者安静留在工作站「待接诊」分组，跨日进搜索/历史，超时未接诊触发「到院未接诊」预警。

**端到端流程**：
```
医生登录 → 今日工作站（含「+ 建档」入口）→
（建档码 / 就诊码 / 手机号搜索 / 快速建档 / 医生端主动建档 / 补录）多入口进入同一份专病档案 →
【弱接诊】患者扫接诊码即资料可见，可直接采集/OCR/AI，不阻塞于"接诊"动作 →
诊间采集（语音接诊 → AI 提取字段；拍照上传 → OCR，含事后/跨日补拍外院病历）→ 看诊前资料汇总 →
手动触发 AI 诊疗辅助 → 医生确认/执行治疗建议 → 字段回写档案 →
生成随访计划 → 同步患者端复诊提醒
```

### 1.3 技术架构

- **前端**：纯 HTML/CSS/JS，单文件 Demo，零依赖构建工具
- **样式**：CSS 自定义属性（`design-system/tokens.css`），医生端复用 `--doctor-*` 变量
- **组件**：自建设计系统，医生端组件前缀 `ds-doctor-`
- **图标**：SVG 线性图标（`.ui-icon` / `ds-doctor-*`），不用 emoji
- **文档图表**：Mermaid
- **版本管理**：Git

### 1.4 关键角色

| 角色 | 实际承担 | 权限 |
|------|---------|------|
| 业务决策人（主任） | 真实门诊场景判断、专病知识库沉淀 | 唯一可确认「待确认事项」 |
| 产品负责人（鱼宝） | 需求口径、范围、验收 | 批准 IN 域内任务 |
| AI 执行层（小鹿） | 文档、Demo、字段、规范 | 边界明确小任务可直接执行 |

## 二、代码规范

### 2.1 文件组织

```
新增页面 → demos/医生端/功能描述.html
修改组件 → design-system/components.css（追加 ds-doctor-* 类名，不改已有）
新增令牌 → design-system/tokens.css（复用 --doctor-*，不新增色值）
新增文档 → docs/01-产品需求/ 或 docs/03-原型与规范/
```

### 2.2 命名风格

| 类别 | 规范 | 示例 |
|------|------|------|
| 医生端组件类名 | `ds-doctor-` 前缀 + 语义化 kebab-case | `ds-doctor-topbar`、`ds-doctor-todo` |
| CSS 变量 | `--doctor-*` 角色前缀 | `--doctor-primary`、`--doctor-orange` |
| 设计令牌 | 复用 `医生端Demo设计令牌_v1.0.md` | 不另起色板 |
| Demo 文件命名 | 医生端_功能描述.html | `医生端_重绘版Demo.html` |
| 状态语义 | `pending`/`collecting`/`draft_ready`/`archived` | 英文枚举，页面用中文 |

### 2.3 注释要求

- Demo HTML 中的 CSS/JS 可用简洁功能注释
- **禁止**在 HTML 页面写产品逻辑说明、设计意图、演示流程、字段 ID、占位备注
- 规则写入 `docs/` 下 Markdown；页面文案只保留医生当下需要看到的内容

### 2.4 禁止引入的新依赖

- **禁止**引入任何 npm 包、CDN 库、外部 CSS 框架
- **禁止**使用 emoji 作为功能图标
- **禁止**使用外部图片链接，素材必须放入 `assets/` 并相对路径引用
- **禁止**将设计系统文件复制到单个 Demo 中——应通过 `<link>` 引用
- **禁止**在医生端使用患者端的青绿渐变或四胶囊底部导航

### 2.5 HTML 文件结构模板

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>痛风智能体 · 医生端 · 页面名称</title>
  <link rel="stylesheet" href="../../design-system/tokens.css">
  <link rel="stylesheet" href="../../design-system/components.css">
  <style>/* 页面特有样式 */</style>
</head>
<body>
  <script>// 页面逻辑（多屏切换 .screen.active）</script>
</body>
</html>
```

## 三、字段与数据规范

### 3.1 字段命名约定

所有字段以患者端字段字典（`docs/02-字段字典/痛风专病智能体患者端字段字典_v1.1.1_20260706.md`）和病历夹 Tab 梳理（`痛风智能体病历夹记录Tab字段梳理_用药门诊_v1.0_20260710.md`）为权威来源，医生端不另造字段。

| 命名约定 | 示例 |
|---------|------|
| 实体前缀 + 下划线 + 字段名 | `patient_profile.name`、`gout_archive.first_pain_time` |
| 枚举值使用小写下划线 | `encounter_status.archived` |
| 状态枚举不直接暴露给患者端 | 页面文案中文，后台英文枚举 |

### 3.2 必填校验规则

| 字段 | 必填 | 校验规则 |
|------|------|---------|
| 医生手机号 (`doctor.mobile`) | 是 | 11 位，验证码登录 |
| 患者姓名 (`patient.name`) | 是（建档） | 非空 |
| 患者性别 (`patient.gender`) | 是（建档） | 男/女 |
| 患者手机号 (`patient.mobile`) | 是（建档，双端可录） | 11 位，双端去重合并 |
| 管理病种 (`relation.disease_code`) | 是 | 建档码带入，患者/医生均不选 |
| 病史/报告/录音 | 否 | 允许跳过，不阻断建档关联 |

### 3.3 类型定义标准

| 类型 | 表示方式 |
|------|---------|
| 文本 | `string` |
| 数字 | `number` / `int` / `float` |
| 日期 | `date`（YYYY-MM-DD） |
| 日期时间 | `datetime`（ISO 8601） |
| 单选 | `enum` |
| 多选 | `enum[]` |
| 文件 | `file`（引用 attachment_file） |
| 布尔 | `boolean` |

### 3.4 核心状态枚举

```javascript
// 建档状态（患者档案）
const ARCHIVE_STATUS = {
  NONE: 'none',              // 未建档
  BASIC_SUBMITTED: 'basic_submitted',  // 建档中
  IN_PROGRESS: 'in_progress',         // 资料补充中
  MANAGED: 'managed',                 // 已纳管
  LOST: 'lost'                        // 已失访
};

// 接诊状态（医生端主线）
const ENCOUNTER_STATUS = {
  PENDING: 'pending',        // 待接诊
  COLLECTING: 'collecting',  // 采集中
  DRAFT_READY: 'draft_ready',// AI 草稿待回写
  ARCHIVED: 'archived'       // 已写入档案
};

// OCR 状态
const OCR_STATUS = {
  NOT_STARTED: 'not_started',
  PROCESSING: 'processing',
  SUCCESS: 'success',
  PARTIAL_SUCCESS: 'partial_success',
  FAILED: 'failed',
  MANUAL_REVIEW: 'manual_review'
};

// 随访状态
const FOLLOWUP_STATUS = {
  NOT_DUE: 'not_due',
  DUE_TODAY: 'due_today',
  OVERDUE: 'overdue',
  AT_RISK: 'at_risk',
  COMPLETED: 'completed',
  RESCHEDULED: 'rescheduled'
};
```

> 已移除旧的 `confirm_status`（candidate/confirmed/corrected/ignored）枚举。患者提交、OCR、录音提取、AI 小结均直接入档，医生随时可改；唯一保留的确认是 **AI 诊疗辅助治疗建议的医生执行确认**（`ai_consult_suggestion.doctor_confirmed`）。

## 四、开发工作流

### 4.1 新增功能标准步骤

1. **分析**：判断属于哪个医生端页面、哪个接诊状态、哪个动作
2. **查规范**：先读 `docs/03-原型与规范/医生端信息降噪与层级优化建议.md` 与 `design-system/医生端Demo设计令牌_v1.0.md`
3. **查复用**：先查 `components.css` 是否有 `ds-doctor-*` 可复用组件
4. **复述确认**（涉及主流程/关键状态/待确认事项时）：
   - 用一小段话复述：要改哪个页面、哪个状态、希望变成什么、哪些内容不要出现
   - **等待用户确认后再动手**
5. **实现**：在 `demos/医生端/` 下创建或修改文件，引用 `design-system/`
6. **自检**：逐项检查是否违反视觉令牌的"禁止"清单（橙色语义、红色按钮等）
7. **更新文档**：新增通用组件或规则，同步更新 `components.css` 或规范
8. **Git 提交**：提交并推送

### 4.2 修改 Bug 标准步骤

1. 定位问题页面和状态
2. 确认是 CSS 样式问题还是交互逻辑问题
3. 边界明确的小任务（修错别字、删内部备注、整理文档）→ **直接执行**
4. 涉及主流程/关键状态的修改 → **先复述再执行**

### 4.3 修改时禁止做的事

- **禁止**改变已有 `ds-doctor-*` 类名的语义（如需新样式，追加新类名）
- **禁止**修改 `tokens.css` 中已有变量值
- **禁止**在页面中新增"开发说明""演示流程""占位备注"
- **禁止**引入新的外部依赖
- **禁止**在医生端使用患者端视觉语言（青绿渐变、大圆角、四胶囊导航）

## 五、避坑指南（绝对不能触碰的红线）

### 5.1 文案红线

| 禁用文案 | 替代表达 | 原因 |
|---------|---------|------|
| 「等待医生确认」（作为按钮） | 「确认」「通过」「驳回」「回写档案」 | 医生端自己是操作方 |
| 「AI 已诊断」 | 「诊疗辅助建议」 | AI 不做医疗诊断 |
| 「AI 已确认」 | 「已整理为可查看资料」 | AI 不替代医生确认 |
| 「OCR 已确认异常」 | 「报告结果仅供参考，以医生诊断为准」 | OCR 直接入档，医生可修正 |
| 「已自动写入正式档案」 | 「已更新档案」 | 患者/AI 内容直接入档 |
| 「扫码签到」 | 「扫码就诊」「扫接诊码」 | 接诊码建立就诊关联 |
| 「预约成功」 | 「复诊提醒已记录」 | 复诊提醒不是预约 |

### 5.2 逻辑红线

1. **手机号补充双端支持**：患者端优先授权，医生端快速建档页也保留手机号录入兜底；双端来源按最近有效录入去重合并
2. **快速建档只能三个字段**：姓名、性别、手机号；年龄/病史/化验/病种选择不放入建档页
3. **病种由建档码带入**，医生端建档页不展示病种选择
4. **患者提交 / OCR / 录音提取 / AI 小结直接写入档案**，医生可随时在接诊/档案编辑时修正覆盖（无确认硬门槛）
5. **AI 诊疗辅助的治疗建议（开药/调药/停药/开检查）必须经医生确认执行**——这是医疗安全硬门槛，不能跳过
6. **建档码与就诊码严格区分**：建档码长期有效（建档关联），就诊码动态生成有时效（本次就诊身份确认）
7. **录音不作为显眼独立模块**：语音接诊入口在患者档案内，录音只是字段来源与证据
8. **医生端引导动作是"接诊患者"，但接诊是弱动作**：详情页主按钮引导接诊，但资料/采集/OCR/AI 辅助不依赖先点接诊；档案是接诊或采集后沉淀的结果，未接诊时资料同样可查可改
9. **医生端主动建档（无码）受支持**：工作站「+ 建档」填姓名/性别/手机号即可建档案，不依赖患者扫任何码；覆盖无码患者/外院转诊/事后补建
10. **OCR 补录支持事后/跨日补拍**：不限于当次接诊中，医生可独立完成后补拍外院病历，报告时间取实际日期

### 5.3 状态流转红线

1. **已纳管患者再扫建档码，不重复创建档案**
2. **过期/无效建档码或就诊码，不创建错误档案或关联**
3. **医生可随时修正档案任何字段**，不需要独立"确认"步骤（治疗决策除外）
4. **AI 草稿字段回写失败，保持草稿态可重试**，不污染已写入档案
5. **患者跳过病史或报告不影响建档关联**，医生端显示资料缺失
6. **弱接诊：患者扫接诊码后 `pending` 状态信息不得丢失**——未点接诊也必须可查、可采集、可跨日检索
7. **`pending` 超阈值必须升级为「到院未接诊」预警**，不能静默沉底导致患者白等

### 5.4 视觉红线（来自 `医生端Demo设计令牌_v1.0.md`）

1. **禁止**使用患者端青绿渐变作为医生端主色——统一蓝色 `#0052d9`
2. **禁止**使用 emoji 或单字占位作为功能图标
3. **禁止**使用大圆角（22px+）和胶囊形按钮——医生端是工作台
4. **禁止**将橙色用于装饰或普通强调——橙色仅限"待确认/待处理"语义
5. **禁止**将红色用于普通按钮——红色仅限"异常/风险/删除"
6. **禁止**在 AI 诊疗辅助卡片省略"诊疗辅助建议"标题
7. **禁止**在医生端使用患者端四胶囊底部导航——改用底部操作条
8. **禁止**页面出现开发说明、字段 ID、内部规则解释、占位备注

### 5.5 数据红线

1. **涉及患者手机号、身份证号、报告图片等敏感信息，Demo 用模拟数据**
2. **字段来源优先级**：医生端编辑/确认 > 患者提交 > OCR > 录音/AI 整理（前者可随时覆盖）
3. **同一字段多来源时以最高优先级为准**，医生端编辑可直接覆盖其他来源
4. **字段字典是字段唯一权威来源**，医生端不另造字段
5. **来源标记六类两端统一**：患者自填/患者上传/录音识别/AI 识别/医生补录/医生确认
