# 痛风专病智能体患者端产品决策日志

> 本文只记录患者端关键产品决策、变更依据与配套验收项，不作为第二份 PRD。当前唯一权威 PRD：`../01-产品需求/痛风专病智能体患者端PRD_v2.0.3.md`。

## 一、产品定位与用户故事

### 1.1 产品定位

痛风专病智能体患者端 V1.0 是痛风专病管理流程中的患者侧小程序入口。它不是独立的健康管理平台，也不是通用问诊工具。本版本重点服务门诊痛风患者的专病建档、资料补充、扫码就诊和纳管后基础管理。

**核心价值主张**：
- 让患者在 **5 分钟内**完成痛风专病快速建档和基础资料提交
- 帮助医生减少重复采集成本
- 帮助患者获得清晰的病历摘要、复诊提醒和后续联系入口

### 1.2 目标用户

| 用户角色 | 特征 | 核心诉求 |
|---------|------|---------|
| 未建档痛风患者 | 首次接触专病管理 | 快速完成建档，了解下一步 |
| 已建档待就诊患者 | 已提交基本信息，等待门诊 | 补充资料，到院扫码就诊 |
| 已纳管患者 | 档案已建立并关联医生 | 查看病历摘要、复诊提醒、联系医生 |
| 复诊患者 | 已纳管且需要再次就诊 | 查看复诊提醒，补充复诊资料 |

### 1.3 用户故事

| 编号 | 用户故事 | 验收标准 |
|------|---------|---------|
| US-01 | 作为未建档患者，我希望扫描医生出示的二维码后快速完成建档，以便进入专病管理 | 扫建档码 → 授权手机号 → 填姓名性别 → 建档成功，3 步完成 |
| US-02 | 作为已建档患者，我希望在候诊时补充痛风病史和化验报告，以便医生接诊时了解我的情况 | 可以逐项填写病史 9 项，每步允许跳过；可以拍照上传报告 |
| US-03 | 作为到院患者，我希望扫描医生的接诊码完成就诊关联，以便医生查看我的资料 | 扫接诊码 → 自动关联 → 看到资料已提交 |
| US-04 | 作为已纳管患者，我希望在首页看到我的病历摘要、复诊提醒和联系医生入口 | 已纳管首页展示病历夹、下次复诊、联系医生 |
| US-05 | 作为复诊患者，我希望收到复诊提醒并提前补充复诊资料 | 复诊到期前收到提醒，可上传复查报告 |
| US-06 | 作为患者，我希望联系我的医生，咨询问题或申请改期 | 可通过留言、发送报告等方式联系医生 |

## 二、功能规格说明

### 2.1 页面清单与交互逻辑

#### 页面 1：手机号补充页

**触发条件**：微信静默登录成功但未获取有效手机号

**交互逻辑**：
1. 展示当前流程说明
2. 优先展示「微信授权手机号」按钮（大按钮，主色）
3. 下方展示「手动输入手机号」入口（次级按钮）
4. 微信授权使用底部弹层样式，展示应用名称、医院科室、授权目的和脱敏手机号；授权带回的号码视为已验证，不重复校验
5. 手动输入先校验格式（11 位、以 1 开头的合法号段），格式通过后提供「获取验证码」，患者接收并回填短信验证码

**数据流转**：
- 输入：微信授权 token，或手动输入手机号 + 短信验证码
- 校验：微信授权号码视为已验证；手动输入必须通过格式校验和短信验证码校验
- 输出：验证通过后 `patient.mobile` 写入患者基础档案；手动输入未通过验证不得写入或进入后续流程

**异常处理**：
- 授权失败：提示重试，保留手动输入入口
- 手机号格式错误（手动输入）：红色提示「手机号格式不正确，请重新输入」
- 未获取验证码（手动输入）：提示「请先获取短信验证码」
- 验证码错误或过期（手动输入）：提示「验证码不正确或已过期，请重新获取」
- 未通过验证仍尝试提交：拦截并提示「请先完成手机号验证」
- 手机号对应多个疑似档案：进入安全提示，不开放患者自行选择

---

#### 页面 2：快速建档页

**触发条件**：手机号已获取，但无痛风专病档案

**交互逻辑**：
1. 展示已扫描的医生信息和专病上下文（如「已扫描王文龙医生建档码，当前进入痛风专病管理」）
2. 表单字段（仅三个）：
   - 姓名：文本输入框，必填
   - 性别：单选（男/女），必填
   - 手机号：自动带入（已在上一页完成），只读展示
3. 协议授权：勾选框，必选
4. 「确认建档」按钮：三个必填项全部完成后可点击

**数据流转**：
- 输入：姓名、性别、手机号
- 建档码携带：`relation.disease_code`、医生信息、机构信息
- 输出：创建 `temporary_archive`，状态设为 `basic_submitted`

**异常处理**：
- 姓名为空：提示「请填写姓名」
- 性别未选：提示「请选择性别」
- 提交失败：保留已填内容，提示重试
- 网络异常：不清空表单

**关键约束**：
- 页面必须在 390×844 视口内一屏展示完成
- 不出现年龄、身份证、病种选择、详细病史等额外字段

---

#### 页面 3：诊前任务页（建档中首页 S2）

**触发条件**：快速建档提交成功

**交互逻辑**：
1. 上方状态栏展示「基本信息已提交」，说明「补充病史和化验报告，方便医生接诊时查看。」
2. 中间信息卡展示：
   - 档案完整度百分比（如「45% 完整度」）
   - 待补资料标签（「痛风病史」「化验报告」）
   - 关联医生信息（卡片底部 footer）
   - 右上角「我的病历夹」入口
3. 主行动按钮：
   - 默认：「补充诊前资料」（进入病史问诊 → 引导上传报告）
   - 资料完成或跳过时改为：「扫码就诊」
4. 次行动：「给医生留言」（联系医生能力未开通时隐藏）
5. 底部快捷胶囊（始终固定）：联系医生 / 上传报告 / 健康打卡 / 饮食问答

**数据流转**：
- 状态判断：`archive.status` + 任务完成情况
- 完整度计算：病史 9 项 + 报告 1 项 = 10 个资料项，按比例计算
- 跳过处理：跳过不影响扫码就诊，仅影响完整度数值

**状态子场景**：
| 子场景 | 机器人消息文案 |
|--------|--------------|
| 建档成功后首次展示 | 「你好，{姓名}，基本信息已提交。接下来可以补充痛风病史和化验报告，医生接诊时会看到这些内容。」 |
| 病史未补充 | 「先来补充一下你的痛风病史吧，医生接诊时可以参考。」 |
| 病史已提交、报告未上传 | 「病史已提交。如果手头有化验报告，可以先上传给医生参考。」 |
| 资料已完成或跳过 | 「资料已整理好。到院后可以扫描医生的接诊码完成就诊。」 |
| 已扫码就诊 | 「已完成扫码就诊，医生接诊后会更新你的档案。」 |

---

#### 页面 4：痛风病史补充页

**触发条件**：患者点击「补充诊前资料」或从诊前任务页进入

**交互逻辑**：
1. 采用引导式分步表单，每次展示一个字段
2. 字段采用口语化提问（如「你还记得第一次痛风大概是什么时候开始的吗？」）
3. 每个字段支持：
   - 单选/多选/日期选择/文本输入（根据字段类型）
   - 「不确定」选项
   - 「跳过」按钮
4. 完成或跳过后进入下一字段
5. 全部字段过完后展示「病史已提交，已更新档案」

**采集字段（9 项）**：

| 序号 | 口语化提问 | 对应字段 | 类型 |
|------|-----------|---------|------|
| 1 | 第一次痛风大概是什么时候开始的？ | `gout_history.first_pain_time` | 日期+填空 |
| 2 | 第一次发作在哪个部位？ | `gout_history.first_attack_site` | 多选+填空 |
| 3 | 身上有痛风石吗？ | `gout_history.has_tophus` | 单选 |
| 4 | 有其他疾病吗？ | `gout_history.comorbidities` | 多选+填空 |
| 5 | 合并症在用什么药？ | `gout_history.comorbidity_medications` | 文本 |
| 6 | 最近关节有没有疼痛或红肿？ | `gout_history.current_pain_status` | 单选+填空 |
| 7 | 在长期吃降尿酸的药吗？ | `gout_history.urate_lowering_medication` | 单选+填空 |
| 8 | 最近在吃消炎止痛药吗？ | `gout_history.anti_inflammatory_medication` | 单选+填空 |
| 9 | 近一年痛风发作了几次？ | `gout_history.attack_count_1y` | 单选+填空 |

**数据流转**：
- 输入：患者逐项填写的字段值
- 输出：直接写入 `gout_archive` 对应字段，医生可随时在接诊时修正

---

#### 页面 5：化验报告上传页

**触发条件**：患者点击「上传化验报告」或从诊前任务页进入

**交互逻辑**：
1. 支持拍照上传和相册选择两种方式
2. 支持连续上传多张报告
3. 上传后展示状态：
   - 上传中：进度提示
   - 识别中：OCR 处理中
   - 识别完成：展示可识别字段摘要
   - 识别失败：提示「报告已上传，医生仍可查看原图」
4. 可继续上传更多报告或完成提交

**数据流转**：
- 上传：创建 `attachment_file` → 创建 `lab_report` → 创建 `ocr_task`
- OCR 成功：回写 `lab_result[]`，`ocr_status = success`，直接写入档案
- OCR 失败：`ocr_status = failed`，原图保留，医生可在接诊时查看原图手动录入

**关键约束**：
- OCR 识别失败不阻断患者扫码就诊
- 报告按报告时间/化验时间归档
- 患者端不把 OCR 识别指标展示为正式诊断结论，提示"结果仅供参考，以医生诊断为准"

---

#### 页面 6：扫码就诊页

**触发条件**：患者点击「扫码就诊」入口

**交互逻辑**：
1. 深色背景、扫描框、动态扫描线
2. 识别到接诊码：浮层显示医生信息和接诊确认
3. 识别到建档码：按建档码规则处理（进入建档流程）
4. 扫码成功 → 建立本次接诊关联 → 进入资料提交状态

**异常处理**：
| 场景 | 处理 |
|------|------|
| 接诊码过期 | 提示「接诊码已过期，请让医生重新出示」 |
| 接诊码无效 | 提示「未识别到有效接诊信息」 |
| 扫到建档码 | 提示「当前是建档码，将进入建档流程」 |
| 患者未建档 | 引导先完成手机号补充和快速建档 |
| 患者已纳管 | 建立本次复诊关联，不重复建档 |

### 2.2 全局交互规则

1. **阿福式对话容器**：患者端所有页面统一使用浅青绿到浅蓝氛围的对话式容器
2. **底部快捷胶囊固定**：联系医生 / 上传报告 / 健康打卡 / 饮食问答，四按钮不因状态变化而增减
3. **跳过不阻断**：病史补充和报告上传允许跳过，不影响扫码就诊
4. **状态恢复**：患者再次进入小程序时，自动恢复到当前状态对应页面
5. **表单保留**：操作失败时不清空已填内容

## 四、决策日志（ADR）

### ADR-001：快速建档只保留三个字段

**日期**：2026-07

**决策**：快速建档页仅包含姓名、性别、手机号三个字段。年龄、病史、用药、化验等进入后续流程。

**原因**：
- 5 分钟建档目标要求表单极简
- 年龄可通过出生日期推算，不需要患者手动填写
- 病史和化验涉及医疗判断，应在诊前资料补充或医生接诊时处理
- 减少患者在建档环节的填写负担和放弃率

**影响**：
- 医生端需在接诊时查看并补充缺失的年龄等信息
- 后续需设计「档案完整度」来追踪资料完善进度

---

### ADR-002：手机号补充以患者端为主，医生端同样需要

**日期**：2026-07

**决策**：手机号补充以患者端（微信授权或手动输入）为优先路径，同时医生端也需具备手机号录入能力，用于覆盖患者未授权或信息缺失的场景。

**原因**：
- 患者端优先：微信授权能力只能在微信环境中调用，且手机号属于患者个人信息，在患者自己设备上授权更符合隐私合规
- 医生端兜底：部分患者可能拒绝微信授权或不会手动输入，此时需要医生在诊间协助补充，避免因手机号缺失导致建档流程中断
- 双端录入需做去重和合并逻辑，以最近一次有效录入为准

**影响**：
- 患者首次进入小程序优先引导完成手机号补充
- 如患者拒绝授权，需提供手动输入备选方案
- 医生端快速建档页需提供手机号录入字段
- 系统需支持手机号双端来源的去重与冲突处理

---

### ADR-002B：手动输入手机号必须完成两层验证

**日期**：2026-07-13

**决策**：微信授权带回的手机号视为已由微信平台完成实名与所有权验证，患者端不再重复校验；患者手动输入手机号时，必须先通过格式校验，再通过短信验证码完成号码所有权校验。两层验证均通过后才允许写入 `patient.mobile` 并进入快速建档、资料补充或接诊关联流程。

**原因**：
- 手动输入存在输错号码、冒用号码和无法触达患者的风险
- 仅检查 11 位格式无法证明号码属于当前患者
- 微信授权与手动输入的可信来源不同，需要采用不同验证策略

**影响**：
- 手动输入流程增加获取验证码、回填验证码、错误与过期处理
- 验证未通过时不得写入患者档案或放行后续流程
- 主 PRD、字段字典、异常状态和验收标准必须保持同一口径

---

### ADR-003：患者提交内容直接写入正式档案，医生保留修正能力

**日期**：2026-07

**决策**：患者提交的病史、报告和 AI/OCR 识别的字段直接写入正式痛风专病档案，不设"必须经医生确认"的硬门槛。医生端保留随时修正/覆盖任何档案字段的能力。

**原因**：
- 降低医生操作负担：去掉独立的"确认"步骤，加快医生接诊流程
- 患者端即时获得感：提交后直接看到档案更新，不需要等待"医生确认"
- 医生修正能力兜底：即使患者填错或 OCR 识别有误，医生在接诊时随时可修改，不需要独立确认环节

**影响**：
- 去掉 `confirm_status` 候选/已确认中间态，患者提交内容与医生记录统一作为档案字段
- 患者端文案从"已提交，待医生确认"改为"已提交，已更新档案"
- 医生端不需要独立的"资料确认工作区"，改为在接诊/档案编辑时直接修改
- 字段来源优先级保留：医生端直接编辑 > 患者提交（后者可被医生随时覆盖）

### ADR-004：建档码和接诊码严格区分

**日期**：2026-07

**决策**：建档码用于引导患者进入专病建档链路，接诊码用于患者到诊后建立本次接诊关联。两个码的流程、文案和后续动作完全不同，不能混用。

**原因**：
- 建档发生在就诊前，接诊发生在到院后，时间窗口不同
- 建档码携带专病上下文（痛风），接诊码携带本次接诊信息
- 混淆会导致流程错误：患者可能用接诊码建档或拿建档码就诊

**影响**：
- 扫码识别后需要判断码类型
- 两个码的页面文案完全不同
- 患者端统一使用「扫码就诊」而不是「扫码签到」

---

### ADR-005：患者端统一「扫码就诊」口径

**日期**：2026-07

**决策**：患者端所有与门诊接诊相关的动作和文案统一使用「扫码就诊」或「扫接诊码」，禁止使用「扫码签到」。

**原因**：
- 「签到」暗示简单打卡，但实际是建立就诊关联关系
- 「就诊」准确表达了扫码的业务目的
- 避免患者以为扫码后即可直接看病，忽略挂号等实际流程

**影响**：
- 所有页面文案审查标准中增加此项
- 复诊提醒页必须提示按医院规则挂号

---

### ADR-006：采用纯 HTML/CSS/JS 的 Demo 技术方案

**日期**：2026-07

**决策**：Demo 阶段采用零依赖的纯静态 HTML 方案，不使用任何前端框架。

**原因**：
- Demo 目标是快速验证产品流程，不需要生产级工程化
- 单个 HTML 文件可以直接在浏览器打开，降低演示门槛
- 避免引入框架后增加维护和学习成本
- 后续正式开发时可迁移到微信小程序框架

**影响**：
- 所有 CSS/JS 内嵌在 HTML 文件中
- 设计系统通过 `<link>` 引用共享的 CSS 文件
- 无热更新、无模块化、无 TypeScript

---

### ADR-007：设计系统从 Demo 中抽取而非预先设计

**日期**：2026-07

**决策**：`design-system/` 目录中的 tokens 和 components 是从已有 Demo 中抽取的稳定样式，而不是预先设计的完整组件库。

**原因**：
- 避免过度设计：Demo 阶段组件需求尚未完全明确
- 确保组件经过实际使用验证，不是纸上谈兵
- 抽取式方法可以随 Demo 迭代自然演进

**影响**：
- `components.css` 中只包含已在 Demo 中验证过的组件
- 新页面优先复用已有组件，确实需要时才新增
- 组件变更需要同步更新 Demo 和规范文档

## 五、验收标准

### 5.1 主流程验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-01 | 建档码进入 | 医生端生成痛风专病建档码 → 患者扫码 → 进入建档流程并自动带出痛风专病上下文 |
| AC-02 | 静默登录 | 患者进入小程序 → 完成微信静默登录 → 无手机号时进入补充流程 |
| AC-03 | 快速建档 | 建档页只包含姓名/性别/手机号，不出现年龄/身份证/病种选择等额外必填项 |
| AC-03B | 手机号验证 | 微信授权号码免重复校验；手动输入号码必须通过格式校验和短信验证码校验，未通过时不得写入档案或进入后续流程 |
| AC-04 | 诊前任务 | 建档后 → 可补充病史 → 可上传报告 → 可扫码就诊 → 每步允许跳过 |
| AC-05 | 扫码就诊 | 扫接诊码 → 建立接诊关联 → 接诊码不与建档码混用 |
| AC-06 | 患者提交直接入档 | 患者提交病史/报告/OCR结果 → 直接写入正式档案 → 医生可在接诊时随时修正 |
| AC-07 | 已纳管 | 档案建立并关联医生后 → 患者端进入已纳管首页 → 展示病历夹/复诊提醒/联系医生 |

### 5.2 关键边界验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-08 | 跳过不阻断 | 患者跳过病史或报告后仍可扫码就诊 |
| AC-09 | 重复建档 | 已纳管患者再扫建档码，不重复创建档案 |
| AC-10 | 过期码处理 | 过期建档码/接诊码，不创建错误档案或关联 |
| AC-11 | 文案审查 | 无「扫码签到」「预约成功」「AI 已诊断」等禁用文案 |
| AC-12 | 状态恢复 | 患者再次进入小程序，恢复到当前状态对应页面 |

### 5.3 数据验收

| 编号 | 验收项 | 验收标准 |
|------|--------|---------|
| AC-13 | 字段字典引用 | 所有字段使用字段字典 v1.1.1 定义的编码和枚举 |
| AC-14 | 来源优先级 | 多来源冲突时按优先级处理：医生端编辑 > 患者提交，后者可被医生随时覆盖 |
| AC-15 | OCR 入档 | OCR 识别结果直接写入档案，低置信度项标注提示，医生可在接诊时修正 |

## 附：跨端说明（患者端 ↔ 医生端）

本 PRD 覆盖**患者端**需求。医生端的建档、接诊、OCR 补拍等流程患者端不直接感知，仅呈现"已更新档案""已纳管"等结果态。医生端相关规格见 [`医生端_PRD_决策日志.md`](./医生端_PRD_决策日志.md)，逻辑基线见 [`INDEX.md`](./INDEX.md)：

- **医生端可主动建档（无码）**：医生在工作站「+ 建档」填姓名/性别/手机号直接建档案，覆盖无码患者/外院转诊/事后补建（ADR-D09）。
- **医生端可事后拍照补录**：接诊结束甚至跨日后，医生可进入患者档案拍照补拍外院病历，OCR 直接入档，不依赖患者在线（ADR-D09）。
- **弱接诊**：患者扫接诊码即资料可见，医生未点「接诊患者」也可采集/OCR/AI 辅助；未接诊患者留「待接诊」分组，超时触发「到院未接诊」预警（ADR-D01）。
