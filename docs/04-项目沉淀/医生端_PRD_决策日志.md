## 一、产品定位与用户故事

### 1.1 产品定位

痛风专病智能体医生端 V1.0 是围绕门诊医生的轻量 MVP，在手机上跑通"医生登录与身份校验 → 今日工作站 → 建档/关联患者 → 诊间采集与沉淀痛风专病档案 → 必要时手动触发 AI 个性化诊疗方案辅助 → 诊后资料沉淀 → 复诊提醒与患者端随访触达"的最小闭环。

它不是完整慢病管理平台，也不是医院 HIS / LIS / 处方系统的替代品。它只补齐医院系统覆盖不足的部分：快速纳管、结构化采集、AI 诊疗辅助、诊后闭环。

**核心价值主张**：
- 让医生在 **5 分钟内**把患者纳入痛风专病管理
- 把主任和专科团队沉淀的诊疗标准转化为下级医生可参考的处理路径
- 把患者说不清、写不清、记不清的信息通过语音、照片和 AI 结构化整理出来

### 1.2 目标用户

| 用户角色 | 特征 | 核心诉求 |
|---------|------|---------|
| 主任医生 | 专科经验权威，负责沉淀标准 | 快速建档、病历拍照补录、复诊提醒 |
| 基层/下级医生 | 面对复杂患者需参考路径 | 手动触发 AI 诊疗辅助、确认执行 |
| 医生助理（规划中） | 协助补录 | 后续版本，一期不做 |

### 1.3 用户故事

| 编号 | 用户故事 | 验收标准 |
|------|---------|---------|

## 二、功能规格说明

### 2.1 页面清单与交互逻辑

#### 页面 1：医生登录页（login）

**触发条件**：首次进入或登录态失效

**交互逻辑**：
1. 手机号输入框（11 位校验）
2. 获取验证码按钮（倒计时）
3. 验证码输入框
4. 「登录」主按钮

**数据流转**：
- 输入：`doctor.mobile` + 验证码
- 后台：校验执业机构、科室、医生身份（`doctor_account`）
- 输出：登录态 → 进入 `workbench`

**异常处理**：
- 验证码错误：提示重试
- 非认证医生：提示无权限（Demo 为模拟）

---

#### 页面 2：今日工作站（workbench）

**触发条件**：登录成功默认进入

**交互逻辑**：
1. 顶部渐变蓝栏：医生姓名 + 科室/职称 + 建档码入口
2. 关键数字卡（3 个）：待接诊 / 复查逾期 / 待回复
3. 主任务卡：「{患者} · 等待接诊」，主按钮「接诊患者」，次按钮「看诊前资料」
4. 今日待办列表（最多 3 条）：动作语言（如"接诊张明""处理陈建国失访预警"）
5. **「待接诊」分组（弱接诊兜底主入口）**：所有已扫接诊码、状态为 `pending` 的患者安静留在此分组，**不超时、不自动消失**，医生空了逐个点开即可；即使未点「接诊患者」，资料也已可查看
6. **「到院未接诊」预警**：患者扫码超过阈值（默认 4 小时或跨日 0 点）仍处 `pending`，从「待接诊」升级为预警色（红），出现在关键数字卡「待接诊」角标 + 今日待办中，提醒医生优先处理，避免患者白跑

**数据流转**：
- 来源：`workbench_task` 聚合（待接诊 + AI 待办 + 逾期触达 + 待回复 + **到院未接诊预警**）
- 优先级：到院未接诊（红）> 风险（红）> 提醒（橙）> 普通（蓝）
- 跨日兜底：当日未接诊患者从「今日待接诊」落入「全部患者 / 历史」，仍可被搜索定位（按姓名/手机号/建档码）

**异常处理**：无任务时展示空态引导

---

#### 页面 3：患者队列（patients）

**触发条件**：从工作站进入或顶部搜索

**交互逻辑**：
1. 搜索框（手机号/姓名）
2. 高密度患者列表行：姓名 + 性别年龄 + 脱敏手机号 + 病程摘要 + 状态 chip + 时间
3. 点击行 → 进入 `detail`

**数据流转**：
- 输入：搜索关键词
- 输出：过滤后的 `patient_profile` 列表
- 状态 chip：待接诊（橙）/ 已写入档案（绿）/ 异常风险（红）

---

#### 页面 4：患者详情 / 专病档案（detail）

**交互逻辑**（接诊前准备页，弱接诊：资料默认可看）：
1. 顶部：姓名、性别年龄、脱敏手机号、当前状态「建档中 · 待接诊」
2. 主按钮「接诊患者」→ 进入 `voice`（诊间采集）；**未点接诊也可直接查看资料、语音采集、OCR、触发 AI 辅助**
3. 三个进度模块：诊前资料 / 化验资料 / 随访计划
4. 更多资料入口（既往史/生活史/家族史/体格检查，不与主线平级）
5. 快捷操作区：「就诊码」按钮（生成本次动态码）、「编辑」按钮（修正档案）

**数据流转**：
- 展示：`patient_profile` + `gout_archive` + 资料进度
- 编辑：医生直接修改任意字段，回写档案（来源标记「医生补录」）

**关键约束**：
- 详情页第一动作仍是「接诊患者」引导，但**弱接诊**下资料、采集、OCR、AI 辅助均不依赖先点接诊——接诊只是计时标记
- 档案默认只展开最关键分组，AI 依据用"查看依据"入口

---

#### 页面 5：语音接诊 / 诊间采集（voice）

**触发条件**：详情页点击「接诊患者」

**交互逻辑**：
1. 顶部：患者信息 + 「接诊中 · 正在整理字段」
2. 中间：采集状态（已识别首次发作时间、待追问用药依从性等）
3. 录音状态轻量展示（"采集中" + 时间），不把"问诊录音"作大标题
4. 底部主按钮「结束接诊，确认字段」

**数据流转**：
- 输入：医生语音问诊
- 处理：AI 转写 + 提取结构化字段（`voice_record`）
- 输出：字段草稿，来源标记「录音识别」，进入待确认资料区
- 同步：录音期间不改变患者端正式档案；医生确认并回写成功后，患者端同步确认后的结果

**异常处理**：
- 录音失败：保留已填，提示重试
- 提取失败：医生手动补录

---

#### 页面 6：补充资料（supplement）

**触发条件**：详情页/AI 辅助页点击「补充资料」；**也支持从工作站「全部患者」进入无码患者档案后补拍**

**交互逻辑**：
1. 拍照上传三类：化验单 / 病历照片 / 药盒照片
2. OCR 识别 → 提取结构化数值/用药
3. 「最近上传」列表，标注 OCR 状态与归档状态
4. 底部「进入资料确认」→ 由医生确认正式归档
5. **事后补拍（关键场景）**：不限于当次接诊中——接诊结束、甚至跨日后，医生都可重新进入该患者档案拍照补录外院病历/历史报告，OCR 结果同样进入待确认资料区

**数据流转**：
- 化验单：OCR → `lab_result[]` 候选字段，来源「AI 识别」
- 病历/药盒：OCR → 用药候选字段，来源「AI 识别」；诊断信息供参考，不自动覆盖已有诊断
- 确认后：医生采纳或修正后写入正式报告/指标；`supplement_material.archive_status = archived`，同步患者端按报告时间归档
- **跨日补拍**：仍写入同一 `gout_archive`，报告时间取拍摄/报告实际日期，不影响历史时间轴

**关键约束**：
- OCR 失败保留原图，医生手动录入
- 低置信度项标注提示，医生可修正
- 事后补拍不要求患者在线或在院，医生独立可完成

---

#### 页面 7：看诊前资料（previsit）

**触发条件**：接诊前或触发 AI 前点击「看诊前资料」

**交互逻辑**（四个模块）：
1. 患者自填问卷：诊前病史/用药/不适，标注提交时间与来源「自填」
2. 外院报告：按报告日期归档，标注格式与来源
3. AI 依据材料：本院化验/检查，展示关键指标与异常值
4. 问诊录音：列表展示，仅作材料，列在底部

底部：「返回档案」+「进入 AI 诊疗辅助」

**数据流转**：汇总 `gout_archive` + `lab_report` + `voice_record` 已有资料，供医生判断完整度

---

#### 页面 8：AI 诊疗辅助（encounter）

**触发条件**：医生手动触发（患者档案入口 或 当前接诊入口）

**交互逻辑**：
1. 顶部标识行：AI 图标 + 「诊疗辅助建议」标题 + 置信度
2. 三段式内容：依据（蓝竖条）/ 风险（红竖条）/ 建议（绿竖条）
3. 证据引用块：列出引用材料
4. 底部：主按钮「确认并执行」+ 次按钮「忽略」

**数据流转**：
- 输入：患者病史/检查/用药/合并症 + 专病知识库
- 输出：`ai_consult_suggestion`（阶段判断/风险/建议方向）
- **硬门槛**：开药/调药/停药/开检查等医疗决策，必须经医生「确认并执行」，`doctor_confirmed = true` 后才成医嘱

**异常处理**：
- 资料不足：提示先补充资料再触发
- 知识库缺失：提示参考通用处理，标注"非专病标准路径"

---

#### 页面 9：资料确认与正式回写（confirm）

**触发逻辑**：接诊结束 AI 生成字段草稿，或医生需要核对多来源

**交互逻辑**：
1. 展示待确认或存在冲突的结构化候选字段：字段名 + 建议值 + 来源 + 状态
2. 医生点击查看依据（诊间片段/患者填写/OCR/历史）
3. 可逐字段采纳/修改/忽略
4. 已一致字段可批量采纳；底部「确认并回写档案」→ 正式写入 `gout_archive`

**数据流转**：
- 输入：待确认候选资料（录音识别/OCR/患者自填/AI 整理）
- 输出：仅采纳或修正后的字段回写档案；来源标记和确认状态保留
- 冲突：多来源不一致时，按优先级展示，医生选最终值

**关键约束（ADR-D03）**：
- 患者提交、OCR、录音提取和 AI 整理均先保存原始材料或候选资料，不得直接写入正式档案
- 医生对字段执行采纳、修正或忽略；只有采纳或修正后才触发正式档案、病历夹和患者端状态更新
- AI 诊疗辅助的治疗建议执行确认仍独立保留（见页面 8）

---

#### 页面 9-bis：本次接诊小结（summary）

**触发逻辑**：接诊结束、事后补录完成或医生主动重新生成

**交互逻辑**：
1. 展示 AI 根据本次语音、报告、处方变化和接诊记录生成的小结草稿
2. 展示生成时间、输入材料和草稿版本，仅医生端可见
3. 医生可编辑、保存草稿，或确认其中需要进入正式档案和患者病历夹的内容
4. 未确认草稿不得覆盖正式病史、用药、检查或处理结果

**数据流转**：
- 输入：本次接诊材料引用与生成时的输入快照
- 输出：`encounter_summary` 草稿版本；医生确认后形成正式回写记录
- 状态：`draft`（草稿）/ `confirmed`（已确认）/ `superseded`（已被新版本替代）

#### 页面 10：建档码（qr）

**交互逻辑**：
1. 展示长期有效的痛风专病建档二维码
2. 文案：「患者扫码后填写姓名、性别和手机号，关联到你的痛风专病档案」
3. 不替代医院叫号系统，不做签到/排队

---

#### 页面 10-bis：医生端主动建档（无码，doctor-create）

**触发条件**：工作站「+ 建档」按钮，或从「全部患者」发现缺档主动补建

**交互逻辑**：
1. 医生填写三个字段：姓名 / 性别 / 手机号（与患者端快速建档一致，保持轻量）
2. 点击「创建档案」→ 直接生成 `patient_profile` + 关联 `gout_archive`，来源标记「医生补录」
3. 建档后进入患者详情页（状态 `pending` / 已纳管，取决于是否已有就诊关联）
4. 不依赖任何码：覆盖无码患者（老人/不会用手机/急诊）、外院转诊现场口述、事后补建

**数据流转**：
- 输入：`patient.name` / `patient.gender` / `patient.mobile`（医生补录，含 ADR-D02 双端去重）
- 输出：创建档案并关联专病；`relation.source = doctor_create`
- 手机号缺失：允许先建档（mock/空），后续用 ADR-D02 双端补录去重

**关键约束**：
- 医生端建档字段严格限制三个，不在此页展开病史/化验（进入采集流程再补）
- 建档前做姓名+手机号查重，命中已有档案则跳转关联而非新建（AC-D03 不重复建档）
- 来源明确标记「医生补录」，追溯与患者自填区分

---

#### 页面 11：就诊码（encounter-qr）

**交互逻辑**：
1. 动态生成本次就诊码，橙色主题区分建档码蓝色
2. 标注「就诊结束后失效」+ 生成时间 + 关联患者
3. 已建档患者到院扫描 → 确认本次就诊身份

**异常处理**：过期就诊码 → 患者端提示重新出示

---

#### 页面 12：复诊 / 逾期提醒（overdue）

**交互逻辑**：
1. 逾期列表（红色标注）+ 即将逾期（橙色）
2. **「到院未接诊」分组（弱接诊兜底预警）**：患者已扫接诊码但超过阈值仍 `pending`，红色标注，提示"到院未接诊，避免患者空等"
3. 结果型待办：逾期人数/已触达/未读未响应/需医生处理
4. 一键批量提醒（AI 先触达，短信兜底）
5. 医生只处理例外

**数据流转**：
- 输入：`followup_plan` 中 `followup_status = overdue`；以及 `encounter_status = pending` 且超过阈值（`arrived_at` 距当前 > 4h 或已跨日）的患者
- 输出：触达记录，状态更新；「到院未接诊」预警可一键跳转详情或批量发短信告知患者稍候

---

### 2.2 全局交互规则

1. **接诊优先**：医生端主线是"接诊患者"，档案是接诊后沉淀结果
2. **多入口同源**：建档码/就诊码/搜索/补录进入同一份档案
3. **来源可追溯**：每个字段带来源标记，医生端呈现
4. **候选资料 + 正式回写**：患者/AI 内容先保存为候选资料；医生采纳或修正后写入正式档案
5. **医疗决策硬门槛**：AI 治疗建议必须经医生确认执行
6. **表单保留**：操作失败不清空已填内容

## 三、数据字典

## 四、决策日志（ADR）

### ADR-D01：医生端主线从"确认档案"转向"接诊患者"，接诊为弱动作

**日期**：2026-07

**决策**：医生端第一眼是"我要接诊谁"，而非"我要看一堆档案"。**但"接诊"是弱动作**——患者扫接诊码建立就诊关联后，资料立即可见，不强制医生先点「接诊患者」才能看资料/做采集。「接诊患者」按钮退化为"标记开始本次看诊"的计时动作，用于统计接诊时长与排队顺序，不再是看资料的门禁。

**原因**：
- 医生真实心智是"现在接诊这个患者"，不是"维护复杂档案"
- 门诊高峰时医生可能来不及逐个点接诊，但患者资料必须立即可查、可处理，否则信息阻塞在"待接诊"里造成漏诊
- 弱接诊保证：忙时不点接诊也不丢信息，空了从「待接诊」分组/搜索/预警三处都能捞回

**影响**：
- 详情页改为接诊前准备页，但**看资料、语音采集、OCR、AI 辅助均可在未点接诊时直接使用**
- 新增诊间采集页、字段确认/回写页
- 档案页改为正式结果页
- 新增「到院未接诊」预警：超过阈值（默认 4 小时或跨日）仍处于 `pending` 的患者进入预警列表，主动提醒医生，避免患者白跑一趟
- 接诊时长从"点击接诊患者"时刻开始计时，而非扫码时刻

### ADR-D02：手机号补充双端支持（复用患者端 ADR-002）

**日期**：2026-07

**决策**：手机号补充以患者端（微信授权/手动）为优先，医生端快速建档页同样保留手机号录入字段作为兜底。双端来源按最近有效录入去重合并。

**原因**：
- 患者端优先符合隐私合规与微信环境能力
- 部分患者拒绝授权或不会输入，需医生诊间兜底
- 避免手机号缺失阻断建档

**影响**：医生端建档页含手机号字段；系统支持双端去重冲突处理

---

### ADR-D03：候选资料经医生确认后正式回写（替代旧 ADR-003）

**日期**：2026-07

**决策**：患者提交的病史/报告、OCR 识别结果、语音提取字段和 AI 整理结果先保存为原始材料或候选资料，不直接写入正式痛风专病档案。医生在「资料确认与正式回写」页对字段采纳、修正或忽略；只有采纳或修正后的字段才写入正式档案，并触发患者端病历夹和状态更新。

**原因**：
- 原始材料即时保存，避免患者、OCR 或录音资料丢失
- 正式医疗档案只使用医生确认后的字段，避免候选内容提前对患者展示
- 医生可对一致字段批量采纳，对冲突字段逐项修正，兼顾效率和可追溯性

**影响**：
- 恢复 `confirm_status`：`candidate`（候选）/ `confirmed`（已采纳）/ `corrected`（修正后确认）/ `ignored`（已忽略）
- 字段确认页是正式档案、病历夹和患者端状态同步的必要节点，不得降级为可选核对页
- 原图、原始回答和原始消息可即时保存；附件保存、OCR 完成和正式报告/指标确认分别记录状态
- 六类来源标记、原始依据、确认状态和确认人/时间只在医生端呈现；患者端仅展示已提交、已确认或已更新等结果

---

### ADR-D04：AI 诊疗辅助治疗建议必须经医生确认执行（医疗安全硬门槛，保留）

**日期**：2026-07

**决策**：AI 诊疗辅助生成的依据/风险/建议为"诊疗辅助建议"，涉及开药、调药、停药、开检查等医疗决策时，必须经医生「确认并执行」，`doctor_confirmed = true` 后才成为医嘱。AI 不直接替医生完成医嘱。

**原因**：
- 医疗安全：治疗决策必须由具备资质的医生执行
- AI 输出基于专病知识库，但是辅助而非替代
- 与 ADR-D03 区分：资料确认决定哪些候选内容进入正式档案；治疗建议执行确认决定是否形成医疗处理记录

**影响**：
- AI 辅助页保留「确认并执行」「忽略」按钮
- 治疗建议未确认不形成医嘱
- 与 ADR-D03 不冲突：病史/用药候选字段经资料确认后可入档；"开药调药"动作还需独立的治疗建议执行确认

### ADR-D05：建档码与就诊码双码分离

**日期**：2026-07

**决策**：建档码长期有效（患者首次扫码建档，关联专病），就诊码本次就诊动态生成有时效（已建档患者到院确认身份）。流程、文案、后续动作不同，不能混用。

**原因**：
- 建档在就诊前，接诊在到院后，时间窗口不同
- 混淆会导致患者用接诊码建档或拿建档码就诊

**影响**：医生端两个独立码屏；患者端统一「扫码就诊」

---

### ADR-D06：来源标记六类用于医生端追溯

**日期**：2026-07

**决策**：医生端专病档案字段、记录条目使用统一六类来源标记（患者自填/患者上传/录音识别/AI 识别/医生补录/医生确认）。

**原因**：
- 医生端需要依据来源和确认状态处理候选资料
- 患者端不展示来源标签、置信度、冲突或内部确认过程，只展示确认后的结果

**影响**：医生端呈现六类来源与确认过程；患者端不展示来源标签

### ADR-D07：纯 HTML/CSS/JS 的 Demo 方案（复用）

**日期**：2026-07

**决策**：Demo 阶段零依赖纯静态 HTML，不使用前端框架。

**原因**：快速验证流程、浏览器直接打开、后续迁移小程序框架

### ADR-D08：设计系统从 Demo 抽取，医生端蓝色工作台（复用）

**日期**：2026-07

**决策**：`design-system/` 从 Demo 抽取稳定样式；医生端复用 `--doctor-*` 变量，不另起色板，保持蓝色高密度工作台风格。

**原因**：避免过度设计，组件经实际验证；两端共享外壳、内部色板不同

### ADR-D09：医生端主动无码建档 + 事后拍照病历识别

**日期**：2026-07

**决策**：
1. **医生端主动建档**：提供「+ 建档」入口，医生填姓名/性别/手机号（三个字段，与患者端快速建档一致）即可直接创建专病档案，不依赖患者扫任何码。覆盖无码患者（老人/不会用手机/急诊）、外院转诊现场口述、事后发现缺档补建。
2. **事后拍照识别**：OCR 补录不限于当次接诊中——接诊结束甚至跨日后，医生可重新进入患者档案拍照补拍外院病历/历史报告；原图即时保存，OCR 结果进入待确认资料区。

**原因**：
- 真实门诊有大量无码场景：老人不会扫码、急诊没带码、外院转诊无档案
- 医生在诊后整理/跨日复盘时才拿到外院纸质病历，需补拍入档
- 降低对"患者必须先扫码"的前置依赖，建档入口掌握在医生手里更稳

**影响**：
- 新增页面 10-bis「医生端主动建档」；工作站增加「+ 建档」入口
- OCR 补录页明确标注"支持事后/跨日补拍"，并进入资料确认与正式回写流程
- 医生端建档来源标记「医生补录」；事后补拍报告时间取实际日期，不污染历史时间轴
- 建档前姓名+手机号查重，命中已有档案则关联不新建（复用 AC-D03）
