# AGENTS.md instructions

<INSTRUCTIONS>
本项目默认使用「痛风产品助手」专家角色。该专家已安装于 WorkBuddy 专家中心，内置项目完整设计规范、需求文档和交互规则。新对话开始时，请从左侧「专家」面板选择「痛风产品助手」激活专家模式；或在 WorkBuddy 中将本工作空间创建为「项目」并绑定该专家为默认专家，此后每次新对话将自动激活。

请默认使用连贯、自然的完整段落回答，少用碎片化短句，除非我明确要求提纲、清单或汇报口径。

涉及主流程、页面信息架构、医生端/患者端关键状态、主任待确认事项、对外展示 Demo 的修改时，先用一小段话复述需求，明确要改哪个角色端、哪个状态、希望变成什么、哪些内容不要出现；等我确认后再修改文件。修错别字、删除内部备注、按已确认规则自检、整理既有文档等边界明确的小任务，可以直接执行并回报结果。

本项目中的 Demo 和页面原型默认按“可直接对外展示”的成品标准处理，不要写给开发者、评审者或自己看的解释性备注。页面文案只保留真实用户或医生当下需要看到的操作信息、状态信息和业务内容；产品逻辑、设计意图、判断规则、演示流程、开发说明、占位备注、内部说明应写入规则文档或对话说明，不要放进界面。

痛风智能体 Demo 的固定规则：患者端采用小程序静默登录；手机号补充发生在登录或扫码进入后的患者端页面，可微信授权或手动输入，不放在医生电脑或建档表单环节；快速建档字段只保留姓名、性别、手机号，年龄进入后续专病档案；医生端外层不要把“问诊录音”作为显眼模块，录音只作为病史采集、化验单冲突、查看 AI 依据、证据来源中的材料出现；痛风病史必须按结构化表单顺序展示分块、字段、填写值和来源，不能只展示一个总答案。患者端诊前任务按“补充痛风病史 → 上传化验报告 → 扫码就诊”串联，每一步允许跳过；患者端不要把“等待医生确认”作为主按钮，不写“扫码签到”，统一使用“扫码就诊/扫接诊码”。

痛风智能体第一期主线默认围绕“快速建档、医患关联、录音提取、OCR 补充、结构化档案、AI 依据、复诊/失访闭环、患者获得感”设计。建档目标是 5 分钟内完成；医生端应在工作台或患者/居民列表外层提供建档入口，不要把建档藏到居民详情深层；OCR 是补充路径，适用于复诊、外院资料、历史资料补录和患者不配合场景，化验单要按报告/化验时间归档；复诊提醒要覆盖提前提醒、逾期提醒、失访列表和医生人工联系；患者端注册后应看到个人档案、科普内容、票令查询、联系医生等有获得感的内容。接诊码由医生端出示，患者端负责扫描；患者端统一使用“扫码就诊”口径，不写“扫码签到”，也不要设计成患者出示接诊码给医生扫。

待与主任确认：患者建档时是否需要手动选择或填写病种。当前原型倾向于由医生端“痛风专病建档码”或医生发起的建档入口自动带出病种，患者只填写姓名、性别、手机号；但需确认真实门诊场景里是否存在一个二维码对应多个专病、患者跨专病建档、或需要患者自行选择病种的情况。确认前，Demo 页面不要把“选择病种”作为必填字段放进快速建档表单。

待与主任确认：患者补充痛风病史、用药和化验资料的使用场景是否默认发生在候诊阶段。当前原型倾向于让患者在扫码建档后、等待医生确认档案前，先自行补充病情、用药和检查资料，适合候诊时提前填充信息，减少医生问诊录入负担；但如果真实流程是在诊间由医生引导患者完成，可能需要另一套交互逻辑，例如医生端发起补充、医生边问边录音并自动回填、患者只确认关键信息，或由医生直接修改结构化字段。明天需要向主任确认：候诊自填是否是一期开通的主路径，诊间录音回填是否作为并行路径或后续路径处理。
</INSTRUCTIONS>
